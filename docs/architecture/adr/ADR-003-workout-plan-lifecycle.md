# ADR-003 Workout Plan Lifecycle를 정의한다

## 문서 정보

* 프로젝트: AI Love Workout
* 문서: ADR-003
* 상태: ACCEPTED
* 작성일: 2026-07-24

---

# 1. 배경

AI Love Workout는 AI가 사용자의 운동 조건을 기반으로 운동 계획을 추천하는 서비스이다.

사용자는 AI가 추천한 운동 계획을 그대로 사용할 수도 있고, 일부 내용을 수정하거나 새로운 추천을 받을 수도 있다.

또한 운동 계획을 확정한 이후 실제 운동을 수행하며 계획과 다른 결과가 발생할 수 있다.

따라서 운동 계획의 생성부터 실제 운동 기록까지의 전체 흐름을 일관된 생명주기로 정의할 필요가 있었다.

---

# 2. 문제

AI가 생성한 운동 계획과 사용자가 실제 수행하는 운동은 동일한 개념이 아니다.

운동 계획 후보, 확정된 운동 계획, 실제 운동 기록을 하나의 객체처럼 관리할 경우 다음과 같은 문제가 발생한다.

* 현재 운동 상태를 명확하게 구분하기 어렵다.
* 후보 수정과 실제 운동 기록이 서로 영향을 줄 수 있다.
* 운동 시작 시점을 정의하기 어렵다.
* 향후 기능 확장 시 데이터 흐름이 복잡해질 수 있다.

---

# 3. 고려한 선택지

## 선택지 1

운동 계획과 실제 운동을 하나의 객체로 관리한다.

### 장점

* 구조가 단순하다.
* 초기 구현이 쉽다.

### 단점

* 운동 계획과 실제 운동을 구분하기 어렵다.
* 계획 수정과 실제 기록이 혼재될 수 있다.
* 상태 관리가 복잡해진다.

---

## 선택지 2

운동 계획의 생명주기를 단계별로 분리한다.

AI 추천 결과를 운동 계획 후보로 생성하고,

사용자가 확인 및 수정한 후 운동 계획을 확정하며,

운동 시작 이후에는 실제 운동 기록을 별도로 관리한다.

### 장점

* 각 단계의 책임이 명확하다.
* 상태 변화가 자연스럽다.
* 실제 운동 기록과 계획을 구분할 수 있다.
* 향후 기능 확장에 유리하다.

### 단점

* 도메인 구조가 다소 복잡해진다.
* 상태 전이를 관리해야 한다.

---

# 4. 최종 결정

운동 계획은 다음과 같은 생명주기를 따른다.

```
AI Recommendation
        │
        ▼
Workout Plan Candidate
        │
        ├──────────────┐
        ▼              ▼
Confirmed Workout Plan   Candidate Rejected
        │              │
        └──────┬───────┘
               ▼
        Workout Session
               │
               ▼
      Exercise Record (0..N)
```

각 단계는 서로 다른 책임을 가지며,

이전 단계의 객체를 다음 단계의 객체와 동일한 것으로 취급하지 않는다.

---

# 5. 결정 이유

운동 계획은 추천 과정과 실제 수행 과정이 명확하게 구분되어야 한다.

사용자는 운동 계획을 수정할 수 있으며,

실제 운동 과정에서는 계획과 다른 운동을 수행할 수도 있다.

생명주기를 분리하면

* 사용자 경험을 단순하게 유지할 수 있고,
* 상태 관리를 명확하게 할 수 있으며,
* AI 추천, 운동 통계, 루틴 추천 등 향후 기능 확장에도 유리하다.

---

# 6. 영향

이 결정으로 인해

* Domain Model은 운동 계획의 생명주기를 반영한다.
* ERD는 각 단계의 관계를 표현할 수 있어야 한다.
* Functional Requirements는 각 단계의 책임을 기준으로 작성한다.
* API는 단계별 상태 전이를 고려하여 설계한다.
