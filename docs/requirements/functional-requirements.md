## 1. 문서 정보

* 프로젝트: AI Love Workout
* Sprint: Sprint 1
* 상태: DRAFT

## 2. 문서 목적

본 문서는 AI Love Workout MVP의 시스템 기능 요구사항을 정의한다.

사용자 스토리를 실제 시스템 기능으로 구체화하며 이후 API, 데이터베이스, UI, 테스트 케이스의 기준 문서로 사용한다.

## 3. 기능 요구사항 작성 원칙

* 모든 기능 요구사항은 하나 이상의 사용자 스토리와 연결한다.
* 기능 요구사항은 사용자 관점이 아니라 시스템 관점으로 작성한다.
* 시스템의 핵심 동작은 “시스템은 ... 해야 한다.” 형식으로 작성한다.
* 하나의 기능 요구사항은 독립적으로 검증할 수 있는 하나의 시스템 기능을 정의한다.
* 사전 조건, 입력, 입력 검증, 처리, 출력과 비즈니스 규칙을 구분한다.
* 아직 결정되지 않은 값이나 정책을 임의로 확정하지 않는다.
* 구현 기술, API 경로와 데이터베이스 구조를 요구사항으로 강제하지 않는다.
* 주요 정상 흐름과 실패 흐름을 검증 시나리오로 정의한다.
* 우선순위는 `mvp-scope.md`와 `user-stories.md`를 따른다.

## 4. 기능 요구사항 ID 규칙

| Prefix | 영역 |
|---|---|
| `FR-AUTH` | 사용자 인증 |
| `FR-USER` | 사용자 프로필 및 기본 운동 정보 |
| `FR-PLAN` | 운동 계획 후보 및 확정 운동 계획 |
| `FR-SESSION` | 운동 세션 |
| `FR-RECORD` | 운동 기록 |
| `FR-NLP` | 자연어 운동 기록 구조화 |
| `FR-HISTORY` | 운동 세션 및 운동 기록 조회 |

기능 요구사항 ID는 다음 형식을 사용한다.

```text
FR-[도메인 코드]-[3자리 순번]
```

예:

* `FR-AUTH-001`
* `FR-PLAN-001`
* `FR-NLP-001`

순번은 각 도메인에서 `001`부터 독립적으로 증가한다.

## 7. 검증이 필요한 사항

* 회원가입 성공 후 자동 로그인 적용 여부
* 회원가입 이메일 형식의 세부 허용 기준
* 비밀번호 길이와 조합 정책
* 닉네임 길이와 허용 문자 정책
* 로그아웃 시 인증 정보 무효화 방식
* Access Token 및 Refresh Token 운영 방식
* 운동 가능한 시간의 최소값과 최대값
* 운동 계획 후보 생성에 사용할 최근 운동 기록의 기간
* 운동 계획 후보에서 수정할 수 있는 필드 범위
* 세트 사이 여러 입력을 최종 운동 기록으로 병합하는 방식
* 자연어 원문과 미완료 구조화 결과의 임시 보존 방식
* 기록 없이 종료된 운동 세션의 기본 조회 화면 노출 방식
* 탈퇴 또는 비활성 사용자 계정의 이메일 재사용 허용 여부

## 8. 기능 요구사항
### 8.1 사용자 인증

#### FR-AUTH-001 이메일 회원가입

##### 우선순위

Must

##### 관련 사용자 스토리

* US-01-01

##### 기능 설명

시스템은 사용자가 이메일, 비밀번호와 닉네임을 입력하여 사용자 계정을 생성할 수 있도록 해야 한다.

시스템은 회원가입 성공 후 사용자에게 회원가입 완료 결과를 제공해야 한다.

회원가입이 완료되면 시스템은
사용자가 로그인하거나 로그인 상태로 전환할 수 있는
후속 인증 흐름을 제공해야 한다.

후속 인증 흐름의 구체적인 방식은
추후 확정한다.

##### 사전 조건

* 없음

##### 입력

* 이메일
* 비밀번호
* 닉네임

##### 입력 검증

* 이메일은 필수 입력값이어야 한다.
* 이메일은 서비스에서 허용하는 이메일 형식이어야 한다.
* 비밀번호는 필수 입력값이어야 한다.
* 비밀번호는 확정된 비밀번호 정책을 만족해야 한다.
* 닉네임은 필수 입력값이어야 한다.
* 닉네임은 확정된 닉네임 정책을 만족해야 한다.

##### 처리

* 시스템은 입력값의 유효성을 검증해야 한다.
* 시스템은 입력한 이메일의 중복 여부를 확인해야 한다.
* 시스템은 원본 비밀번호를 그대로 저장하지 않아야 한다.
* 시스템은 인증에 사용할 수 있도록 보호된 형태로 비밀번호를 저장해야 한다.
* 시스템은 사용자 계정을 생성해야 한다.

##### 출력

* 회원가입 성공 여부
* 생성된 사용자 계정을 식별할 수 있는 정보
* 회원가입 완료 결과

##### 비즈니스 규칙

* 현재 가입되어 있는 이메일과 동일한 이메일로 새로운 사용자 계정을 생성할 수 없어야 한다.
* 입력값 검증 또는 이메일 중복 확인이 실패하면 사용자 계정을 생성하지 않아야 한다.
* 회원가입 성공 응답에는 비밀번호 또는 보호된 비밀번호 값을 포함하지 않아야 한다.
* 회원가입 실패 시 사용자가 수정할 수 있는 입력 문제를 안내해야 한다.

##### 예외 및 실패 조건

* 필수 입력값이 누락된 경우 회원가입을 처리하지 않아야 한다.
* 이메일 형식이 유효하지 않은 경우 회원가입을 처리하지 않아야 한다.
* 비밀번호 정책을 만족하지 않는 경우 회원가입을 처리하지 않아야 한다.
* 닉네임 정책을 만족하지 않는 경우 회원가입을 처리하지 않아야 한다.
* 입력한 이메일이 이미 사용 중인 경우 사용자 계정을 생성하지 않고 중복 이메일임을 안내해야 한다.
* 사용자 계정 생성 처리에 실패한 경우 불완전한 사용자 계정이 남지 않아야 한다.

##### 검증 시나리오

```text
Given 입력한 이메일로 가입된 사용자 계정이 존재하지 않고
And 사용자가 유효한 이메일, 비밀번호와 닉네임을 입력했을 때
When 회원가입을 요청하면
Then 사용자 계정이 생성되어야 한다.
And 원본 비밀번호가 그대로 저장되어서는 안 된다.
And 사용자에게 회원가입 완료 결과가 제공되어야 한다.
And 로그인하거나 로그인 상태로 전환할 수 있는 후속 인증 흐름이 제공되어야 한다.
```

```text
Given 입력한 이메일로 가입된 사용자 계정이 이미 존재할 때
When 동일한 이메일로 회원가입을 요청하면
Then 새로운 사용자 계정을 생성하지 않아야 한다.
And 이메일이 이미 사용 중임을 안내해야 한다.
```

```text
Given 사용자가 유효하지 않은 회원가입 입력값을 제공했을 때
When 회원가입을 요청하면
Then 사용자 계정을 생성하지 않아야 한다.
And 수정이 필요한 입력값을 안내해야 한다.
```

#### FR-AUTH-002 로그인

##### 우선순위

Must

##### 관련 사용자 스토리

* US-01-02

##### 기능 설명

시스템은 가입된 사용자가 이메일과 비밀번호를 이용해 인증 상태를 생성할 수 있도록 해야 한다.

시스템은 인증된 사용자가 자신의 기능과 데이터에 접근할 수 있도록 해야 한다.

##### 사전 조건

* 없음

##### 입력

* 이메일
* 비밀번호

##### 입력 검증

* 이메일은 필수 입력값이어야 한다.
* 이메일은 서비스에서 허용하는 이메일 형식이어야 한다.
* 비밀번호는 필수 입력값이어야 한다.

##### 처리

* 시스템은 입력값의 유효성을 검증해야 한다.
* 시스템은 입력한 이메일에 대응하는 사용자 계정을 조회해야 한다.
* 시스템은 입력한 비밀번호와 저장된 보호 비밀번호의 일치 여부를 검증해야 한다.
* 시스템은 인증 성공 시 사용자를 식별할 수 있는 인증 상태를 생성해야 한다.

##### 출력

* 로그인 성공 여부
* 인증이 필요한 기능에 사용할 인증 정보
* 로그인한 사용자를 식별할 수 있는 정보

##### 비즈니스 규칙

* 이메일과 비밀번호가 모두 일치하는 경우에만 인증 상태를 생성해야 한다.
* 로그인 실패 시 인증 상태를 생성하지 않아야 한다.
* 로그인 실패 응답은 사용자 계정의 존재 여부를 과도하게 노출하지 않아야 한다.
* 로그인 성공 응답에는 원본 비밀번호 또는 보호된 비밀번호 값을 포함하지 않아야 한다.
* 인증 정보는 다른 사용자의 데이터에 접근하는 데 사용할 수 없어야 한다.

##### 예외 및 실패 조건

* 필수 입력값이 누락된 경우 로그인을 처리하지 않아야 한다.
* 이메일 형식이 유효하지 않은 경우 로그인을 처리하지 않아야 한다.
* 사용자 계정이 존재하지 않는 경우 로그인에 실패해야 한다.
* 비밀번호가 일치하지 않는 경우 로그인에 실패해야 한다.
* 인증 상태 생성에 실패한 경우 로그인 성공으로 처리하지 않아야 한다.

##### 검증 시나리오

```text
Given 가입된 사용자가 올바른 이메일과 비밀번호를 입력했을 때
When 로그인을 요청하면
Then 인증 상태가 생성되어야 한다.
And 사용자는 인증이 필요한 자신의 기능과 데이터에 접근할 수 있어야 한다.
```

```text
Given 사용자가 잘못된 이메일 또는 비밀번호를 입력했을 때
When 로그인을 요청하면
Then 인증 상태를 생성하지 않아야 한다.
And 계정 존재 여부를 과도하게 노출하지 않는 오류를 안내해야 한다.
```

```text
Given 사용자가 필수 로그인 입력값을 누락했을 때
When 로그인을 요청하면
Then 로그인 처리를 진행하지 않아야 한다.
And 수정이 필요한 입력값을 안내해야 한다.
```

#### FR-AUTH-003 JWT 기반 인증

##### 우선순위

Must

##### 관련 사용자 스토리

* US-01-02
* US-01-03

##### 기능 설명

시스템은 로그인에 성공한 사용자를 식별하고 인증이 필요한 기능의 접근 여부를 판단하기 위해 JWT 기반 인증 정보를 사용해야 한다.

##### 사전 조건

* 없음

##### 입력

* 로그인에 성공한 사용자 식별 정보
* 인증이 필요한 요청에 포함된 JWT 기반 인증 정보

##### 입력 검증

* 인증 정보는 시스템이 발급한 JWT 형식이어야 한다.
* 인증 정보에는 원본 비밀번호와 보호된 비밀번호를 포함하지 않아야 한다.
* 유효하지 않은 인증 정보는 유효한 인증 정보로 처리하지 않아야 한다.

##### 처리

* 시스템은 로그인 성공 시 JWT 기반 인증 정보를 생성해야 한다.
* 시스템은 인증이 필요한 요청에서 JWT 기반 인증 정보를 검증해야 한다.
* 시스템은 검증된 인증 정보를 통해 요청 사용자를 식별해야 한다.
* 시스템은 인증에 실패한 요청의 보호 기능 접근을 허용하지 않아야 한다.

##### 출력

* 로그인 성공 시 생성된 JWT 기반 인증 정보
* 인증 요청의 성공 또는 실패 결과
* 검증된 사용자 식별 결과

##### 비즈니스 규칙

* 다른 사용자의 JWT 기반 인증 정보를 이용해 해당 사용자의 데이터에 접근할 수 없어야 한다.
* 원본 비밀번호나 보호된 비밀번호를 JWT에 포함하지 않아야 한다.
* JWT의 구체적인 종류, 발급 구성, 유효기간과 갱신 방식은 인증 및 보안 설계에서 확정한다.
* 로그아웃 시 JWT 기반 인증 정보를 제한하는 구체적인 방식은 인증 및 보안 설계에서 확정한다.

##### 예외 및 실패 조건

* 인증 정보가 누락된 경우 인증이 필요한 기능에 접근할 수 없어야 한다.
* 인증 정보가 유효하지 않은 경우 인증이 필요한 기능에 접근할 수 없어야 한다.
* 인증 정보에서 사용자를 식별할 수 없는 경우 인증에 실패해야 한다.
* JWT 기반 인증 정보 생성에 실패한 경우 로그인 성공으로 처리하지 않아야 한다.

```text
Given 사용자가 올바른 이메일과 비밀번호로 로그인에 성공했을 때

Then JWT 기반 인증 정보가 생성되어야 한다.
And 인증 정보에는 원본 비밀번호와 보호된 비밀번호가 포함되지 않아야 한다.
```

```text
Given 사용자가 유효한 JWT 기반 인증 정보를 가지고 있을 때
When 인증이 필요한 자신의 기능에 접근하면
Then 시스템은 인증 정보를 검증해야 한다.
And 요청 사용자를 식별할 수 있어야 한다.
```

```text
Given JWT 기반 인증 정보가 없거나 유효하지 않을 때
When 인증이 필요한 기능에 접근하면
```

#### FR-AUTH-004 로그아웃

##### 우선순위

Must

##### 관련 사용자 스토리

* US-01-03

##### 기능 설명

시스템은 로그인한 사용자가 현재 인증 상태를 종료할 수 있도록 해야 한다.

시스템은 로그아웃으로 종료된 인증 정보를 이용한 보호 기능 접근을 허용하지 않아야 한다.

##### 사전 조건

* 없음

##### 입력

* 현재 사용자의 인증 정보
* 로그아웃 요청

##### 입력 검증

* 로그아웃 요청의 인증 정보가 현재 사용자를 식별할 수 있어야 한다.
* 인증 정보가 없거나 유효하지 않은 경우 인증된 로그아웃 요청으로 처리하지 않아야 한다.

##### 처리

* 시스템은 로그아웃 대상 인증 상태를 식별해야 한다.
* 시스템은 현재 인증 상태를 종료해야 한다.
* 시스템은 로그아웃 이후 기존 인증 정보의 사용을 제한해야 한다.

* 인증 상태 종료 결과

* 사용자는 자신의 인증 상태만 종료할 수 있어야 한다.
* 로그아웃 이후 종료 대상 인증 정보로 보호된 기능에 접근할 수 없어야 한다.
* 로그아웃 처리 방식은 최종 인증 구조와 일치해야 한다.
* 인증 정보 무효화의 구체적인 방식은 인증 및 보안 설계에서 확정한다.

##### 예외 및 실패 조건

* 인증되지 않은 사용자가 로그아웃을 요청한 경우 새로운 인증 상태를 만들거나 사용자 데이터를 변경하지 않아야 한다.
* 유효하지 않은 인증 정보로 로그아웃을 요청한 경우 인증된 사용자의 요청으로 처리하지 않아야 한다.
* 인증 상태 종료 처리에 실패한 경우 로그아웃 성공으로 안내하지 않아야 한다.

```text
Given 로그인한 사용자가 유효한 인증 정보를 가지고 있을 때
When 로그아웃을 요청하면
Then 현재 인증 상태가 종료되어야 한다.
And 종료 대상이 된 기존 인증 정보로 인증이 필요한 기능에 접근할 수 없어야 한다.
```

```text
Then 새로운 인증 상태를 생성하지 않아야 한다.
```

### 8.2 운동 계획 후보 및 확정 운동 계획

#### FR-PLAN-001 운동 계획 후보 생성

##### 우선순위

Must

##### 관련 사용자 스토리

* US-03-01
* US-03-02
* US-03-03
* US-03-10

##### 기능 설명

시스템은 인증된 사용자가 오늘 운동 가능한 시간과 하나 이상의 운동 유형을 기준으로 운동 계획 후보 생성을 요청할 수 있도록 해야 한다.

시스템은 사용자의 운동 목표, 운동 경험 수준, 사용 가능한 장비와 최근 운동 기록을 조회하여 운동 계획 생성 규칙에 적용해야 한다.

시스템은 요청 1건에 대해 운동 계획 후보 1개를 생성해야 한다.

##### 사전 조건

* 없음

##### 입력

* 오늘 운동 가능한 시간
* 하나 이상의 운동 유형

##### 입력 검증

* 오늘 운동 가능한 시간은 필수 입력값이어야 한다.
* 오늘 운동 가능한 시간은 확정된 허용 범위 안의 값이어야 한다.
* 하나 이상의 운동 유형이 선택되어야 한다.
* 선택한 운동 유형은 러닝, 웨이트 또는 맨몸운동 중 하나 이상이어야 한다.

##### 처리

* 시스템은 요청 사용자를 식별해야 한다.
* 시스템은 사용자의 운동 목표를 조회해야 한다.
* 시스템은 사용자의 운동 경험 수준을 조회해야 한다.
* 시스템은 사용 가능한 운동 장비를 조회해야 한다.
* 시스템은 운동 계획 후보 생성에 사용할 최근 운동 기록을 조회해야 한다.
* 시스템은 입력값과 조회한 정보를 운동 계획 생성 규칙에 적용해야 한다.
* 시스템은 가용 시간과 장비 조건을 만족하는 운동 계획 후보를 생성해야 한다.
* 최근 운동 기록이 없는 경우에는 기본 정보를 참고하여 운동 규칙에 따라 운동 계획 후보를 생성한다.
* 운동 계획 후보를 생성할 수 없는 경우에는 운동 계획 후보를 저장하지 않는다.

##### 출력

* 운동 계획 후보 생성 성공 여부
* 생성된 운동 계획 후보 한 개

##### 비즈니스 규칙

* 시스템은 요청 한 건에 대해 운동 계획 후보 한 개를 생성해야 한다.
* 운동 계획 후보의 예상 총 소요 시간은 사용자가 입력한 운동 가능 시간을 초과하지 않아야 한다.
* 운동 계획 후보에는 사용자가 선택하지 않은 운동 유형을 임의로 포함하지 않아야 한다.
* 운동 계획 후보는 사용자가 선택한 모든 운동 유형을 포함해야 한다.
* 운동 계획 후보에는 사용자가 사용할 수 없는 장비가 필요한 운동을 포함하지 않아야 한다.
* 운동 계획의 핵심 조건과 수행 내용은 운동 규칙에 따라 결정해야 한다.
* 생성형 AI가 전체 운동 계획을 임의로 결정하지 않아야 한다.
* 생성된 결과는 사용자가 선택하거나 수정하여 확정하기 전까지 확정 운동 계획으로 처리하지 않아야 한다.
* 최근 운동 기록이 없는 신규 사용자의 경우 기본 정보를 참고하여 운동 규칙에 따라 운동 계획 후보를 생성한다.

##### 예외 및 실패 조건

* 필수 입력값이 누락되거나 유효하지 않은 경우 운동 계획 후보를 생성하지 않아야 한다.
* 운동 계획 후보 생성에 필요한 사용자 기본 정보가 없는 경우 후보를 생성하지 않아야 한다.
* 선택한 모든 운동 유형을 사용 가능한 시간 안에 구성할 수 없는 경우 후보를 생성하지 않아야 한다.
* 선택한 운동 유형과 사용 가능한 장비 조건에 맞는 운동 종목이 없는 경우 후보를 생성하지 않아야 한다.
* 운동 계획 후보를 생성할 수 없는 경우 그 이유와 변경 가능한 조건을 사용자에게 안내해야 한다.
* 운동 계획 후보 생성 처리에 실패한 경우 불완전한 후보를 성공 결과로 제공하지 않아야 한다.

##### 검증 시나리오

```text

And 운동 계획 후보 생성에 필요한 기본 정보를 설정했고
And 오늘 운동 가능한 시간과 하나 이상의 운동 유형을 유효하게 입력했을 때
When 운동 계획 후보 생성을 요청하면
Then 시스템은 사용자 기본 정보와 최근 운동 기록을 운동 계획 생성 규칙에 적용해야 한다.
And 사용자가 입력한 운동 가능 시간을 초과하지 않는 운동 계획 후보 한 개를 생성해야 한다.
And 운동 계획 후보는 사용자가 선택한 모든 운동 유형을 포함해야 한다.

```

```text
When 사용자가 운동 계획 후보 생성을 요청하면
Then 운동 계획 후보를 생성하지 않아야 한다.
```

#### FR-PLAN-002 복합 운동 시간 배분

##### 우선순위

Must

##### 관련 사용자 스토리

* US-03-04
* US-03-10

##### 기능 설명

시스템은 사용자가 한 개 이상의 운동 유형을 선택한 경우 선택한 각 운동 유형에 예상 배정 시간을 산정해야 한다.

시스템은 운동 유형별 예상 배정 시간과 운동 전환 시간 및 휴식 시간을 계산하여 예상 총 소요 시간을 산정해야 한다.

##### 사전 조건

* 없음

##### 입력

* 오늘 운동 가능한 시간
* 한 개 이상의 선택된 운동 유형

##### 입력 검증

* 오늘 운동 가능한 시간은 유효한 값이어야 한다.
* 선택된 운동 유형은 한 개 이상이어야 한다.
* 선택된 각 운동 유형은 서비스에서 지원하는 운동 유형이어야 한다.
* 선택된 운동 유형에는 동일한 운동 유형이 중복되어서는 안 된다.

##### 처리

* 시스템은 사용자가 선택한 운동 유형을 확인해야 한다.
* 시스템은 운동 계획 생성 규칙을 이용하여 각 운동 유형의 예상 배정 시간을 산정해야 한다.
* 시스템은 두 개 이상의 운동 유형이 선택된 경우 운동 계획 생성 규칙에 따라 운동 전환 예상 시간을 산정해야 한다.
* 시스템은 계획에 포함되는 휴식 시간을 산정해야 한다.
* 시스템은 운동 유형별 배정 시간, 운동 전환 시간과 휴식 시간을 합산하여 예상 총 소요 시간을 계산해야 한다.
* 시스템은 계산된 예상 총 소요 시간이 사용자가 입력한 운동 가능 시간을 초과하는지 확인해야 한다.
* 시스템은 단일 운동 유형이 선택된 경우에도 운동 가능 시간을 초과하지 않는 범위에서 운동 유형 별 배정 시간과 예상 총 소요시간을 산정한다.
* 시스템은 선택한 모든 운동 유형을 가용 시간 안에 배분할 수 없는 경우 조건 변경이 필요함을 안내해야 한다.

##### 출력

* 운동 유형별 예상 배정 시간
* 운동 전환 예상 시간
* 휴식 예상 시간
* 전체 예상 총 소요 시간
* 시간 배분 결과

##### 비즈니스 규칙

* 두 개 이상의 운동 유형이 선택된 경우 선택된 각 운동 유형에 예상 배정 시간이 포함되어야 한다.
* 각 운동 유형의 배정 시간이 0보다 커야 한다.
* 운동 유형별 배정 시간, 운동 전환 시간과 휴식 시간의 합은 사용자가 입력한 운동 가능 시간을 초과하지 않아야 한다.
* 예상 총 소요 시간은 사용자가 입력한 운동 가능 시간과 반드시 동일할 필요는 없으며 초과하지 않아야 한다.
* 시스템은 사용자가 선택한 운동 유형을 시간 부족을 이유로 임의로 제외하지 않아야 한다.
* 시간 배분의 구체적인 산정 방식은 운동 계획 생성 규칙에서 정의한다.
* 산정된 운동 유형별 예상 배정 시간, 운동 전환 예상 시간, 휴식 예상 시간과 예상 총 소요 시간은 운동 계획 후보에 포함되어야 한다.
* 단일 운동 유형을 선택한 경우에도 운동 규칙에 따라 예상 배정 시간을 산정해야 한다.

##### 예외 및 실패 조건

* 오늘 운동 가능한 시간이 유효하지 않은 경우 시간 배분을 수행하지 않아야 한다.
* 선택된 운동 유형이 유효하지 않은 경우 시간 배분을 수행하지 않아야 한다.
* 선택된 모든 운동 유형에 유효한 예상 배정 시간을 할당할 수 없는 경우 운동 계획 후보를 생성하지 않아야 한다.
* 예상 총 소요 시간이 사용자가 입력한 운동 가능 시간을 초과하는 경우 운동 계획 후보를 생성하지 않아야 한다.
* 시간 배분에 실패한 경우 변경 가능한 조건을 안내해야 한다.
* 시간 배분 처리에 실패한 경우 불완전한 배분 결과를 성공 결과로 제공하지 않아야 한다.

```text
Given 인증된 사용자가 유효한 운동 가능 시간을 입력했고
And 두 개 이상의 운동 유형을 선택했을 때
When 시스템이 복합 운동 계획의 시간을 배분하면
Then 선택된 각 운동 유형에 0보다 큰 예상 배정 시간이 포함되어야 한다.
And 두 개 이상의 운동 유형에 대한 운동 전환 예상 시간이 계산되어야 한다.
And 운동 계획에 포함되는 휴식 예상 시간이 계산되어야 한다.
And 예상 총 소요 시간은 운동 유형별 예상 배정 시간, 운동 전환 예상 시간과 휴식 예상 시간의 합으로 계산되어야 한다.
And 예상 총 소요 시간은 사용자가 입력한 운동 가능 시간을 초과하지 않아야 한다.
And 산정된 시간 배분 결과는 운동 계획 후보에 포함되어야 한다.
```
