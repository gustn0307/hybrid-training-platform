# 오늘 논의한 내용

오늘은 AI Love Workout의 요구사항 문서와 ADR을 대상으로 전체 검토를 진행하였다.

검토 목적은 현재 상태에서 Architecture 단계로 넘어갈 수 있는지 판단하고, 문서 간 충돌과 기능 누락을 기준으로 다음 작업 순서를 결정하는 것이었다.

Codex 리뷰 결과 상위 요구사항 문서의 연결은 대체로 양호했지만 Functional Requirements에서 운동 세션, 운동 기록, 자연어 구조화와 기록 조회 영역의 MVP Must 기능이 대량으로 누락되어 있음을 확인하였다.

또한 운동 계획 후보 재생성에 대해 상위 문서에서 확정한 정책과 FR-PLAN-006의 기존 문장이 직접 충돌하고 있음을 확인하였다.

오늘은 남은 시간을 고려하여 FR-PLAN-008 작성을 시작하지 않고, FR-PLAN-006의 정책 충돌을 먼저 해결한 뒤 Study, Meeting, Journal을 정리하고 커밋·푸시하기로 하였다.

# 주요 논의 사항

## 1. Architecture 진입 가능 여부

현재 문제 정의, 페르소나, 사용자 여정, 사용자 스토리와 MVP Scope는 대체로 일관되게 연결되어 있다.

그러나 다음 MVP Must 영역에 대응하는 Functional Requirements가 작성되지 않았다.

* 운동 세션 시작, 조회와 종료
* 진행 중 세션 중복 방지
* 운동 기록이 없는 세션 종료
* 웨이트, 맨몸운동과 러닝 기록 저장
* 운동 기록 수정과 삭제
* 자연어 입력, 구조화, 확인, 수정과 저장
* 자연어 구조화 실패와 미완료 입력 처리
* 날짜별 운동 세션과 운동 기록 조회

따라서 전체 Domain Model, ERD와 API를 확정하는 Architecture 단계로 바로 넘어가지 않기로 하였다.

---

## 2. 운동 계획 후보 재생성 정책 충돌

상위 문서에서는 운동 계획 후보 재생성을 다음과 같이 정의하고 있다.

* 현재 운동 계획 후보가 존재해야 한다.
* 기존 후보 생성 시 사용한 운동 가능 시간과 운동 유형을 유지한다.
* 최신 운동 목표, 경험 수준, 사용 가능한 장비와 최근 운동 기록을 다시 조회한다.
* 시간 또는 운동 유형을 직접 변경하는 흐름은 재생성으로 처리하지 않는다.
* 재생성 성공 시에만 새 후보가 현재 후보가 된다.
* 실패 시 기존 현재 후보를 유지한다.
* 기존 후보와 추천 이유는 보존한다.
* 자동 확정이나 운동 세션 시작을 수행하지 않는다.

기존 FR-PLAN-006에는 시간, 운동 유형과 장비 변경을 재생성 입력 조건 변경으로 해석할 수 있는 문장이 남아 있었다.

이 문장은 동일한 재생성 기능을 서로 다른 방식으로 구현하게 만들 수 있으므로 즉시 수정하기로 하였다.

---

## 3. FR-PLAN-006 책임 경계 수정

FR-PLAN-006에서 다음 정책을 확정하였다.

* 오늘 운동 가능한 시간은 후보 수정 항목이 아니다.
* 선택한 운동 유형은 후보 수정 항목이 아니다.
* 시간 또는 운동 유형을 변경하려면 FR-PLAN-001에 따른 새로운 후보 생성 흐름을 시작한다.
* 사용 가능한 운동 장비는 후보 수정 요청에서 직접 변경할 수 없다.
* 기본 운동 정보의 장비가 변경되어도 기존 후보는 자동으로 변경되지 않는다.
* 이후 생성 또는 재생성에서 최신 장비 정보를 다시 조회한다.
* 후보 수정 과정에서 새로운 후보 생성이나 재생성을 자동으로 요청하지 않는다.
* 후보 수정은 확정 운동 계획 생성이나 운동 세션 시작을 발생시키지 않는다.

---

## 4. FR-PLAN-008 작업 시점

FR-PLAN-008은 다음 정책을 하나의 요구사항으로 검토해야 한다.

* 재생성 가능 조건
* 기존 시간과 운동 유형 유지
* 최신 사용자 정보와 최근 기록 재조회
* 새로운 추천 이유 생성
* 성공 시 현재 후보 교체
* 실패 시 기존 현재 후보 유지
* 이전 후보 보존
* 이전 후보 조작 금지
* 자동 확정과 자동 세션 시작 금지

남은 시간 안에 작성, 검토와 충돌 수정을 모두 끝내기 어렵다고 판단하였다.

따라서 FR-PLAN-008은 다음 작업의 첫 번째 독립 작업으로 진행하기로 하였다.

그러나 먼저 Functional Requirements를 완성한 뒤 ADR을 최신 정책에 맞게 갱신하는 순서가 더 안정적이라고 판단하였다.

# 결정 사항

* 현재 상태에서는 전체 Architecture를 확정하지 않는다.
* MVP Must 기능의 Functional Requirements를 먼저 완성한다.
* 시간 또는 운동 유형 변경은 후보 재생성이 아니라 새로운 후보 생성으로 처리한다.
* 장비 변경은 기본 운동 정보 수정으로 처리하고 기존 후보를 소급 변경하지 않는다.
* FR-PLAN-006 수정은 오늘 완료한다.
* FR-PLAN-008은 다음 작업에서 작성과 검토를 하나의 단위로 진행한다.
* 오늘은 Study, Meeting과 Journal을 정리한 뒤 커밋·푸시한다.
