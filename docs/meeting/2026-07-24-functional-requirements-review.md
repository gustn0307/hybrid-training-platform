# 오늘 논의한 내용

오늘은 운동 계획 기능 요구사항(FR-PLAN-001~006)에 대해 여러 차례 검토를 진행하였다.

주요 목표는 기능 요구사항을 완성하는 것이 아니라, 각 기능의 책임과 정책을 명확하게 정의하여 일관된 설계를 만드는 것이었다.

특히 ChatGPT와 Codex의 리뷰를 반복하며 기능 간 경계가 모호한 부분과 테스트 가능한 요구사항인지 여부를 중심으로 검토하였다.

---

# 주요 논의 사항

## 1. 운동 계획 후보(Edit)와 재생성(Regenerate)의 책임

운동 계획 후보를 수정하는 기능과 새로운 추천을 생성하는 기능은 서로 다른 책임을 가지므로 별도의 기능으로 관리하기로 하였다.

운동 종목 삭제, 순서 변경, 세트 수 변경 등은 기존 후보를 수정하는 기능이며,

운동 가능 시간, 운동 유형, 사용 가능한 장비 변경은 새로운 추천 조건이므로 재생성 기능으로 분리하기로 하였다.

---

## 2. 운동 계획 후보의 선택 정책

동일한 운동 계획 후보는 한 번만 확정할 수 있도록 정책을 정의하였다.

이미 확정된 후보를 다시 선택하는 경우에는 새로운 운동 계획을 생성하지 않고 요청을 실패 처리하기로 하였다.

이를 통해 중복된 확정 운동 계획 생성과 데이터 불일치를 방지한다.

---

## 3. 후보 수정의 영향 범위

운동 계획 후보를 수정하더라도

* 자동으로 운동 계획이 확정되지 않는다.
* 자동으로 운동 세션이 시작되지 않는다.
* AI 추천 결과 자체가 다시 생성되지 않는다.

후보 수정은 기존 후보의 내용을 변경하는 기능으로 한정하기로 하였다.

---

## 4. 원자적(Atomic) 수정 정책

여러 항목을 동시에 수정하는 요청은 모두 성공하거나 모두 실패하도록 처리하기로 하였다.

일부 항목만 수정되는 경우에는 사용자에게 예측하기 어려운 결과를 만들 수 있으므로 허용하지 않는다.

---

## 5. 기능 요구사항 작성 원칙

기능 요구사항에서는

* 구현 기술
* 데이터베이스 구조
* API 구현 방식

보다

* 시스템의 책임
* Business Rule
* 테스트 가능한 동작

을 중심으로 작성하기로 하였다.

또한 Domain Model이나 ADR에서 다루어야 할 내용이 기능 요구사항에 포함되지 않도록 문서의 역할을 구분하기로 하였다.

---

# 결정 사항

* 운동 계획 후보 수정과 재생성은 별도 기능으로 관리한다.
* 동일한 운동 계획 후보는 한 번만 확정할 수 있다.
* 후보 수정은 자동 확정이나 운동 시작을 발생시키지 않는다.
* 다중 수정 요청은 원자적으로 처리한다.
* 기능 요구사항은 시스템의 책임과 정책을 중심으로 작성한다.
