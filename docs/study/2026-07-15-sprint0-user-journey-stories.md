# Sprint 0 사용자 여정과 사용자 스토리 학습

## 오늘 학습한 내용

### 사용자 여정

내가 이해한 의미:

사용자 여정은 추후 사용자 스토리와 기능 요구사항 등을 작성하기 전에 사용자가 어떤 상황에서 서비스를 사용하기 시작하고, 어떤 순서로 행동하는지를 시간순으로 기록하는 문서이다.

각 단계에서 사용자의 행동뿐 아니라 생각, 감정, 페인 포인트와 서비스가 제공할 수 있는 기회도 함께 기록한다.

#### 피드백 및 보완할 부분

전반적으로 잘 이해했다.

다만 사용자 여정은 단순히 사용자 스토리와 기능 요구사항을 작성하기 위한 사전 문서만은 아니다. 사용자 흐름, UI/UX, 예외 처리, 서비스 기회와 검증할 가설을 발견하는 기준으로도 사용한다.

또한 사용자 여정은 화면 목록과 다르다.

화면 목록은 어떤 화면이 필요한지를 나열하지만, 사용자 여정은 사용자가 어떤 상황에서 왜 행동하고, 그 과정에서 무엇을 생각하고 불편해하는지를 표현한다.

---

### 사용자 스토리

내가 이해한 의미:

사용자 스토리는 사용자 여정을 더욱 구체화하여 각 행동을 세분화하고, 세분화된 행동에서 사용자 가치, 우선순위와 성공 조건 등을 정의하여 추후 기능 요구사항을 작성하기 위한 문서이다.

AI Love Workout을 예로 들면 오늘 운동 가능한 시간을 입력하려는 사용자의 행동이 있다면 다음과 같은 사용자 스토리를 작성할 수 있다.

> 나는 운동 시간이 불규칙한 사용자로서 오늘 운동 가능한 시간을 입력하고 싶다. 왜냐하면 주어진 시간 안에 완료할 수 있는 운동 계획 후보를 받고 싶기 때문이다.

이 행동이 서비스에 반드시 필요한지 우선순위를 정하고, 인수 조건을 Given-When-Then 방식으로 작성하여 구현 완료 여부와 테스트 기준을 명확히 한다.

#### 피드백 및 보완할 부분

핵심 개념을 잘 이해했다.

다만 사용자 스토리는 사용자가 원하는 기능이나 불편한 점을 단순히 기록하는 문서라기보다 다음 세 가지를 표현하는 단위이다.

* 누가 사용하는가
* 무엇을 원하는가
* 왜 원하는가

사용자 스토리의 중심은 기능이 아니라 사용자가 얻으려는 가치이다.

예를 들어 “시간 입력 기능이 필요하다”보다 “제한된 시간 안에 완료할 수 있는 운동 계획을 받고 싶다”가 사용자 가치에 더 가깝다.

또한 인수 조건은 테스트를 편하게 만드는 용도뿐 아니라 개발자, 기획자와 테스트 담당자가 같은 완료 기준을 공유하게 하는 역할도 한다.

---

### Epic과 사용자 스토리

두 개념의 차이:

Epic은 사용자에게 제공하려는 큰 기능 영역이나 사용자 가치의 묶음이고, 하나의 Epic은 여러 개의 사용자 스토리로 나눌 수 있다.

사용자 스토리는 Epic을 사용자가 실제로 경험할 수 있고 개별적으로 완료 여부를 확인할 수 있는 크기로 나눈 단위이다.

AI Love Workout의 `EPIC-06 자연어 운동 기록 구조화`에는 다음과 같은 사용자 스토리가 포함된다.

* 자연어 운동 결과 입력
* 여러 운동 구조화
* 구조화 결과 확인
* 구조화 결과 수정
* 구조화 결과 저장
* 자연어 구조화 실패 시 일반 입력
* 미완료 운동 기록 입력 처리

#### 피드백 및 보완할 부분

기존 답변의 “Epic은 큰 가지”라는 이해는 방향이 맞지만, 조금 더 정확하게는 하나의 사용자 스토리로 구현하거나 검증하기에는 너무 큰 사용자 가치 또는 기능 영역이다.

Epic과 사용자 스토리의 차이는 단순히 크기뿐 아니라 검증 가능성에도 있다.

Epic 전체는 한 번에 완료 여부를 판단하기 어렵지만, 개별 사용자 스토리는 인수 조건을 통해 독립적으로 완료 여부를 확인할 수 있어야 한다.

---

### 사용자 스토리와 개발 작업

두 개념의 차이:

사용자 스토리는 사용자의 관점에서 어떤 결과와 가치를 얻고 싶은지를 정의한다.

개발 작업은 해당 사용자 스토리와 인수 조건을 구현하기 위해 개발자가 수행해야 하는 프론트엔드, 백엔드, 데이터베이스와 테스트 작업이다.

예를 들어 사용자 스토리가 다음과 같을 수 있다.

> 사용자는 운동 가능한 시간을 입력하여 해당 시간 안에 완료할 수 있는 운동 계획 후보를 받고 싶다.

이 스토리를 구현하기 위한 개발 작업은 다음처럼 나뉠 수 있다.

* 운동 가능 시간 입력 UI 구현
* 입력값 검증 구현
* 운동 계획 후보 생성 API 구현
* 운동 가능 시간 저장 또는 전달 구조 구현
* 가용 시간을 초과하지 않는지 검증하는 테스트 작성

#### 피드백 및 보완할 부분

사용자 스토리에는 사용 과정과 감정 전체를 모두 기록하기보다 사용자 역할, 원하는 행동과 얻고 싶은 가치를 중심으로 작성한다.

사용자의 상황, 감정과 전체 행동 순서는 사용자 여정에서 더 자세히 다룬다.

또한 개발 작업은 단순히 개발하는 과정이라는 의미를 넘어서 사용자 스토리를 구현 가능한 기술 단위로 분해한 작업이다.

하나의 사용자 스토리는 여러 개발 작업으로 분해될 수 있으며, 하나의 개발 작업이 여러 사용자 스토리에 공통으로 사용될 수도 있다.

### 인수 조건

사용자 스토리에 인수 조건이 필요한 이유:

인수 조건은 사용자 스토리가 어떤 조건에서 어떤 행동을 했을 때 어떤 결과를 제공해야 완료된 것으로 판단할 수 있는지를 정의한다.

인수 조건을 작성하면 개발 과정에서 구현 범위를 명확하게 할 수 있고, 테스트 과정에서도 기능이 성공적으로 동작하는지 객관적으로 확인할 수 있다.

예를 들어 기록 없는 운동 세션 종료는 다음 세 가지 흐름으로 나뉜다.

* 사용자가 기록 없는 세션의 종료를 요청한다.
* 안내를 확인하고 종료한다.
* 안내를 확인한 뒤 종료를 취소한다.

각 행동의 결과가 다르므로 별도의 Given-When-Then 조건으로 작성해야 한다.

#### 피드백 및 보완할 부분

잘 이해했다.

특히 인수 조건은 정상 동작만 정의하는 것이 아니라 잘못된 입력, 취소, 실패와 권한 문제도 정의해야 한다.

인수 조건이 없으면 같은 사용자 스토리를 두고도 개발자마다 완료 상태를 다르게 해석할 수 있다.

예를 들어 한 개발자는 종료 버튼만 동작하면 완료됐다고 생각할 수 있지만, 다른 사람은 기록 없는 세션의 통계 제외까지 구현되어야 완료됐다고 판단할 수 있다.

인수 조건은 이러한 해석 차이를 줄이는 기준이다.

---

### Happy Path와 예외 흐름

두 흐름을 모두 정의해야 하는 이유:

Happy Path는 사용자가 유효한 값을 입력하고 시스템도 정상적으로 동작하여 목표를 완료하는 대표적인 정상 흐름이다.

예외 흐름은 사용자의 실수나 변심, 시스템 장애, 유효하지 않은 입력 또는 진행 상태 충돌 등으로 정상 흐름을 그대로 완료하지 못하는 상황이다.

정상 흐름만 정의하면 실제 사용 중 발생할 수 있는 문제에 대응하기 어렵기 때문에 두 흐름을 모두 정의해야 한다.

AI Love Workout의 예외 흐름에는 다음과 같은 사례가 있다.

* 자연어 구조화 실패 후 일반 입력 화면으로 전환
* 운동 기록이 없는 세션 종료
* 미완료 입력이 있는 상태에서 세션 종료 요청
* 진행 중인 운동 세션이 있는데 새로운 세션 시작 요청
* 운동 계획 후보를 생성할 수 없는 경우

#### 피드백 및 보완할 부분

전반적으로 잘 이해했다.

다만 “개발 과정에서 구현의 어려움으로 인한 변경”은 일반적으로 예외 흐름이라기보다 요구사항 변경 또는 기술적 제약에 가깝다.

예외 흐름은 주로 사용자나 시스템이 정상 경로를 벗어났을 때 제품이 어떻게 동작할지를 정의한다.

## AI Love Workout에 적용할 내용

* 사용자 여정을 운동 전, 운동 중, 운동 후의 흐름으로 연결한다.
* 운동 기록이 없는 세션은 사용자 확인 후 종료할 수 있게 하되 운동 통계와 다음 운동 계획 후보 생성에서는 제외한다.
* 사용자 스토리 ID를 기능 요구사항, API와 테스트에 연결하여 추적할 수 있게 한다.
* 운동 중의 빠른 기록과 운동 종료 전 여러 운동 결과의 일괄 입력을 모두 지원하는 흐름을 설계한다.
* AI가 구조화한 운동 결과는 사용자가 확인하고, 필요한 경우 수정한 뒤 저장을 확정하기 전에는 최종 운동 기록으로 저장하지 않는다.
* 자연어 구조화가 실패해도 일반 입력을 통해 운동 기록을 남길 수 있게 한다.

## 내가 아직 헷갈리는 내용

* 세트 사이에 여러 번 입력한 내용을 어떤 방식으로 하나의 운동 기록으로 합칠지 헷갈린다.
* 세트별 입력 문제를 DB 정규화만으로 해결할 수 있는지, 먼저 운동 기록의 저장 단위와 도메인 모델을 결정해야 하는지 추가 학습이 필요하다.
* 인증이 만료되었을 때 작성 중인 자연어 입력과 구조화 결과를 어디까지 임시 보존해야 하는지 헷갈린다.
* 사용자 여정이나 사용자 스토리가 너무 커졌을 때 별도 문서로 나눌지, Epic 또는 더 작은 스토리로 분해할지 판단하는 기준이 아직 부족하다.
* 운동 계획 후보에서 사용자가 수정할 수 있는 필드 범위를 어디까지 허용해야 하는지 헷갈린다.
* 작성 중인 자연어 입력을 브라우저, 서버 또는 Redis 중 어디에 임시 보존해야 하는지는 기술 설계 단계에서 학습이 필요하다.

#### 피드백 및 보완할 부분

첫 번째 질문의 경우 DB 정규화가 바로 정답은 아니다.

먼저 다음을 결정해야 한다.

* 운동 기록의 저장 단위가 운동 종목 단위인지
* 세트 단위의 상세 기록이 필요한지
* 각 세트의 중량과 반복 횟수가 다를 때 어떻게 표현할지
* 자연어 입력 여러 건을 언제 하나의 최종 기록으로 합칠지

이 도메인 규칙을 정한 뒤 그 결과를 관계형 데이터 모델로 표현할 때 정규화를 적용한다.

즉, 정규화는 요구사항을 결정하는 방법이라기보다 결정된 데이터 구조의 중복과 이상 현상을 줄이기 위한 데이터베이스 설계 방법이다.

## 오늘 내가 직접 결정한 것

* 대표 사용자 여정: 사용자는 퇴근 후 AI Love Workout에 접속하여 오늘 운동 가능한 시간과 원하는 운동 유형을 입력하고, 운동 계획 후보를 확인한 뒤 선택·수정·거절한다. 이후 운동 세션을 시작하고 운동 중 또는 운동 종료 후 결과를 기록한 뒤 세션을 종료하고 저장된 운동 내역을 확인한다.
* 운동 전 핵심 행동: 운동 가능한 시간을 입력하고 하나 이상의 운동 유형을 선택한 뒤 운동 계획 후보와 추천 이유를 확인하고 선택·수정·거절한다.
* 운동 중 핵심 행동: 맨몸운동과 웨이트는 세트 종료 후 휴식 시간에 결과를 기록할 수 있으며, 기록 입력을 중단하고 즉시 진행 중인 운동 세션으로 돌아갈 수도 있다.
* 운동 후 핵심 행동: 운동 중 기록하지 않은 결과를 자연어로 일괄 입력하고, 구조화 결과를 확인하거나 수정한 뒤 저장한다. 이후 운동 세션을 종료하고 날짜별 세션과 운동 기록을 조회한다.
* 가장 중요한 예외 흐름: 자연어 구조화가 실패해도 일반 입력으로 운동 기록을 저장할 수 있어야 한다. 운동 기록이 없는 세션도 사용자 확인 후 종료할 수 있지만 운동 통계와 다음 운동 계획 후보 생성용 최근 운동 기록에서는 제외한다.
* 사용자 스토리 ID 규칙: `US-[Epic 번호]-[Story 번호]`
* 사용자 스토리 ID 예시: `US-06-07`

#### 피드백 및 보완할 부분

기존 대표 사용자 여정에 있던 “통계를 쌓아 나의 운동일지를 만든다”는 표현은 장기 통계 기능이 MVP 범위로 오해될 수 있어, 현재 MVP에서 지원하는 날짜별 운동 세션과 기록 조회 중심으로 조정했다.

또한 운동 전 사용자는 개별 운동 종목이 아니라 먼저 러닝, 웨이트와 맨몸운동 같은 운동 유형을 선택한다. 따라서 운동 전 핵심 행동에서는 “운동 종목 선택”보다 “운동 유형 선택”으로 표현하는 것이 현재 용어 기준과 일치한다.

## 페르소나에서 사용자 스토리로 연결한 예시

### 페르소나의 핵심 문제

퇴근 후 운동 가능한 시간이 일정하지 않고, 제한된 시간 안에서 러닝과 근력운동을 어떻게 배분할지 결정하기 어렵다.

### 사용자 목표

사용자는 오늘 확보한 운동 시간을 입력하고 해당 시간 안에서 러닝과 근력운동을 어떻게 나누어 수행할지 빠르게 결정하고 싶다.

### 사용자 스토리

나는 운동 시간이 불규칙한 사용자로서 오늘 운동 가능한 시간을 입력하고 싶다. 왜냐하면 주어진 시간 안에 완료할 수 있는 운동 계획 후보를 받고 싶기 때문이다.

### 인수 조건

```text
Given 로그인하고 기본 설정을 완료한 사용자가
When 오늘 운동 가능한 시간을 입력하면
Then 입력한 시간이 운동 계획 후보 생성 조건에 포함되어야 한다.
```

```text
Given 사용자가 허용되지 않는 시간 값을 입력했을 때
When 후보 생성을 요청하면
Then 요청을 처리하지 않아야 한다.
And 수정이 필요한 입력값을 안내해야 한다.
```

#### 피드백 및 보완할 부분

사용자 목표에는 전체 서비스 이용 순서를 모두 넣기보다 페르소나의 핵심 문제에서 직접 이어지는 결과를 중심으로 작성하는 것이 좋다.

기존 사용자 목표에는 계획 선택, 운동 수행, 자연어 기록과 기록 조회까지 모두 포함되어 있어 하나의 목표라기보다 전체 사용자 여정에 가까웠다.

이번 연결에서는 다음처럼 한 가지 흐름에 집중했다.

```text
운동 시간이 불규칙함
→ 제한된 시간에 맞는 계획이 필요함
→ 오늘 운동 가능한 시간 입력 스토리
→ 입력 시간이 계획 생성 조건에 포함됨
```

## 오늘의 한 문장

오늘 배운 내용을 한 문장으로 정리:

사용자 여정과 사용자 스토리를 구체적으로 작성하는 과정은 오래 걸리고 고려할 예외도 많지만, 사용자의 문제와 기능의 이유, 완료 기준을 구현 전에 명확히 하여 실제 개발 과정의 혼란과 재작업을 줄이는 기준을 만드는 일이다.

#### 개인적인 느낌

요구사항을 작성하는 과정이 생각보다 오래 걸리고 고려할 내용도 많아서 단순히 코딩하는 과정이 더 재미있다고 느꼈다.

하지만 이번 작업을 통해 요구사항을 충분히 고민하지 않고 바로 코딩하면 구현 중에 더 많은 충돌과 재작업이 생길 수 있다는 점을 이해했다.

## 사용자 여정 초안 검토에서 발견한 점

* 제공받은 사용자 여정 초안은 가설이며, 내가 실제 사용 상황을 생각해 작성한 답변과 비교해 수정해야 한다.
* 러닝은 운동 종료 후 기록하는 것이 자연스럽지만, 맨몸운동과 웨이트는 세트 종료 후 휴식 시간에 기록하는 것이 자연스러울 수 있다.
* 운동 중 기록과 운동 종료 후 일괄 기록은 서로 배타적인 방식이 아니라 상황에 따라 함께 제공할 수 있는 흐름이다.
* 자연어 기록이 오래 걸리면 세트 사이 휴식과 다음 운동 흐름을 방해할 수 있으므로 기록 속도와 조작 단계가 중요하다.
* 운동 계획 후보 재생성은 사용자에게 필요한 행동이지만 현재 MVP에서는 Should로 관리한다.
* 운동 계획 후보를 선택하거나 수정한 결과가 확정 운동 계획이며, “확정 계획 후보”라는 용어는 사용하지 않는다.

## 사용자 행동과 데이터 구조의 차이

사용자가 세트가 끝날 때마다 기록하고 싶다는 요구가 있다고 해서 데이터베이스에 세트별 행을 반드시 저장해야 하는 것은 아니다.

사용자 여정에서는 세트 사이에 빠르게 기록할 수 있어야 한다는 행동과 UX 요구를 정의한다.

세트별 상세 기록을 별도로 저장할지, 운동 종목별 요약 기록으로 저장할지는 기능 요구사항과 데이터베이스 설계 단계에서 결정한다.

---

## 기록 없는 운동 세션 처리에서 배운 점

* 사용자가 운동 세션을 시작했더라도 실수, 일정 변경 또는 기록 의사 부족으로 운동 기록 없이 종료할 수 있다.
* 기록 없는 세션을 강제로 막기보다 종료 전에 통계와 추천에 반영되지 않는다는 사실을 안내하고 사용자가 결정하게 하는 방식이 적절하다고 판단했다.
* 운동 세션의 종료 상태와 운동 기록의 존재 여부는 별개의 정보이다.
* DB에 저장된 데이터라고 해서 모든 데이터가 운동 통계나 AI 추천에 사용되는 것은 아니다.
* 기록 없이 종료된 세션의 조회 방식은 사용자 여정에서 확정하지 않고 기능 요구사항과 UI 설계 단계에서 결정한다.

---

## 기록 없는 세션 결정이 다른 문서에 미치는 영향

사용자 여정에서 기록 없는 운동 세션 종료를 허용하기로 결정하면서 기존 README와 MVP 범위의 운동 세션 정의도 수정해야 했다.

기존에는 운동 세션이 하나 이상의 운동 기록을 포함한다고 정의했지만, 현재는 운동 기록이 없는 세션도 사용자 확인 후 종료하고 보존할 수 있다.

다만 기록 없는 세션은 실제 운동 수행을 확인할 수 없으므로 운동 통계와 다음 운동 계획 후보 생성용 최근 기록에서 제외한다.

이를 통해 데이터의 저장 여부와 통계 및 추천에 사용하는 여부가 서로 다른 기준이라는 점을 배웠다.

---

## 승인된 요구사항 변경에서 배운 점

사용자 여정을 구체화하면서 운동 기록 없이 세션을 종료하는 흐름이 필요하다고 결정했다.

이 결정은 사용자 여정 문서에만 영향을 주는 것이 아니라 README의 용어 정의와 승인된 MVP 범위에도 영향을 주었다.

따라서 승인된 문서라도 새로운 결정과 충돌하면 수정하고 최종 수정일과 변경 이유를 남겨야 한다.

또한 세션의 시작 시각부터 종료 시각까지의 값은 실제 운동 시간을 보장하지 않으므로 총 운동 시간이 아니라 총 세션 시간으로 표현하는 것이 더 정확하다.

---

## 사용자 여정에서 사용자 스토리로 연결하며 배운 점

사용자 여정에 중요한 사용자 행동이나 예외 흐름이 추가되면 사용자 스토리에도 연결되어야 한다.

운동 중 자연어 입력과 운동 종료 후 일괄 입력은 처리 과정이 같으므로 하나의 사용자 스토리에서 입력 시점의 차이로 정의했다.

반면 저장되지 않은 입력이 있는 상태에서 세션을 종료하는 흐름은 계속 작성, 입력 폐기와 종료 취소라는 독립적인 사용자 선택이 있으므로 별도 사용자 스토리로 분리했다.

사용자 스토리는 화면이나 API 개수보다 사용자에게 제공되는 가치와 검증 가능한 행동을 기준으로 나누어야 한다.

---

## 사용자 스토리와 MVP 범위의 관계

미완료 운동 기록 입력 처리 사용자 스토리를 추가하면서 해당 기능이 승인된 MVP 범위에 존재하는지 확인해야 했다.

사용자 스토리에 Must 기능을 추가했지만 `mvp-scope.md`에 해당 기능이 없으면 사용자 스토리가 상위 범위를 임의로 확장한 것이 된다.

이번 프로젝트에서는 저장되지 않은 자연어 입력이나 구조화 결과가 사용자 확인 없이 유실되거나 저장되는 것을 방지하는 기능이 기록 신뢰성에 중요하다고 판단하여 MVP Must 범위에 반영했다.

---

## 사용자 여정과 사용자 스토리의 추적 범위

사용자 여정의 모든 문장을 사용자 스토리로 변환할 필요는 없다.

서비스 접속 단계의 핵심 가치 확인, 사용자의 기대와 감정 등은 사용자 경험을 이해하기 위한 내용이지만 반드시 독립적인 MVP 기능은 아니다.

반면 운동 순서 선택, 기록 입력 중단 후 세션 복귀와 같이 시스템의 동작이 필요한 행동은 기존 사용자 스토리의 인수 조건에 연결해야 한다.

또한 인증 만료 시 입력 보존처럼 필요성은 있지만 구체적인 처리 방식이 결정되지 않은 내용은 Must 스토리로 바로 확정하지 않고 검증이 필요한 사항으로 관리한다.

---

## 요구사항 추적성에서 배운 점

요구사항은 각각 독립적으로 작성되는 문서가 아니라 다음처럼 연결된다.

```text
페르소나의 핵심 문제
→ 사용자 여정의 행동과 페인 포인트
→ 사용자 스토리의 가치와 인수 조건
→ 기능 요구사항
→ API와 데이터 모델
→ 테스트 케이스
```

상위 문서에서 새로운 결정을 내리거나 하위 문서에서 새로운 Must 기능을 발견하면 관련 문서 전체에 미치는 영향을 확인해야 한다.

이를 통해 구현할 기능이 왜 필요한지, 어떤 사용자 문제에서 시작되었는지와 어떤 조건을 만족해야 완료되는지 추적할 수 있다.
