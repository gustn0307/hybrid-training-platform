# 오늘 학습한 내용

## 전체 문서 리뷰는 Architecture 진입 가능 여부를 판단하는 작업이다.

개별 문서가 잘 작성되어 있더라도 문서 사이에서 기능이 누락되거나 정책이 충돌하면 안정적인 Domain Model, ERD와 API를 만들기 어렵다.

이번 리뷰에서는 다음 연결을 기준으로 문서 간 추적성을 확인하였다.

* Problem Definition
* Persona
* User Journey
* User Story
* MVP Scope
* Functional Requirements
* ADR

상위 문서의 완성도만 보고 Architecture로 넘어가는 것이 아니라, MVP Must User Story가 실제 Functional Requirements로 충분히 구체화되었는지 확인해야 한다는 점을 배웠다.

---

## Critical과 Major는 구현 결과를 바꾸는 문제이다.

이번 Codex 리뷰에서 Critical로 판단된 문제는 다음과 같다.

* 운동 세션, 운동 기록, 자연어 구조화와 기록 조회 영역의 Must FR 누락
* 상위 재생성 정책과 FR-PLAN-006의 직접 충돌

이 문제들은 단순한 문장 개선이 아니라 구현 책임과 데이터 구조를 다르게 만들 수 있다.

따라서 Architecture보다 먼저 해결해야 한다.

반면 용어 통일, 관련 문서 링크, 추적성 매트릭스 등은 중요하지만 핵심 정책을 먼저 완성한 뒤 정리할 수 있다.

---

## 후보 수정, 재생성과 신규 생성은 서로 다른 Command이다.

오늘 확정한 구분은 다음과 같다.

* 후보 수정: 현재 후보 안의 수정 가능한 항목을 사용자가 직접 변경한다.
* 후보 재생성: 기존 후보 생성 시 사용한 운동 가능 시간과 운동 유형은 유지하고 최신 운동 프로필과 최근 운동 기록을 다시 조회하여 새로운 후보를 생성한다.
* 신규 후보 생성: 운동 가능 시간 또는 선택한 운동 유형을 변경하여 새로운 조건으로 후보를 생성한다.

따라서 운동 가능 시간이나 선택한 운동 유형을 변경하려는 요청은 후보 수정이나 재생성으로 처리하지 않고 FR-PLAN-001의 새로운 운동 계획 후보 생성 흐름으로 처리한다.

---

## 장비 설정 변경은 기존 후보를 소급 변경하지 않는다.

사용 가능한 장비는 운동 계획 후보의 직접 수정 항목이 아니다.

사용자가 기본 운동 정보에서 장비를 변경하더라도 이미 생성된 현재 후보의 내용은 자동으로 변경되지 않는다.

이후 새로운 후보를 생성하거나 재생성할 때 최신 장비 정보를 다시 조회하여 반영한다.

이를 통해 과거에 생성된 후보의 의미를 유지하고, 기본 운동 정보 변경과 후보 수정의 책임을 분리할 수 있다.

---

## 시간 제한이 있을 때는 완료 가능한 작업 단위를 선택해야 한다.

FR-PLAN-008은 재생성 성공과 실패, 현재 후보 교체, 이전 후보 보존, 조작 금지와 추천 이유 생성까지 함께 검토해야 하는 요구사항이다.

남은 시간이 짧은 상태에서 작성을 시작하면 검토와 충돌 수정까지 끝내지 못하고 미완료 상태로 남을 가능성이 크다.

따라서 오늘은 이미 수정한 FR-PLAN-006을 기준으로 문서 기록과 커밋·푸시까지 완료하고, 다음 작업에서 FR-PLAN-008을 독립된 단위로 작성하기로 하였다.

# AI Love Workout에 적용한 내용

FR-PLAN-006에 다음 정책을 반영하였다.

* 오늘 운동 가능한 시간과 선택한 운동 유형은 후보 수정 항목이 아니다.
* 시간 또는 운동 유형을 변경하여 새로운 후보가 필요하면 FR-PLAN-001의 생성 흐름을 시작한다.
* 사용 가능한 장비는 후보 수정 요청에서 직접 변경할 수 없다.
* 기본 운동 정보의 장비 변경은 기존 후보를 변경하지 않는다.
* 이후 후보 생성 또는 재생성 시 최신 장비 정보를 다시 조회한다.
* 후보 수정은 새로운 후보 생성, 재생성, 자동 확정 또는 운동 세션 시작을 발생시키지 않는다.
* 후보 수정 후에도 최초 추천 이유를 유지하며 수정 결과 전체에 대한 새로운 추천 이유로 취급하지 않는다.

# Codex 리뷰에서 배운 점

Codex 전체 리뷰를 통해 상위 요구사항의 일관성이 높더라도 Functional Requirements의 작성 범위가 부족하면 Architecture 준비도가 충분하지 않을 수 있음을 확인하였다.

또한 ACCEPTED 상태의 ADR도 이후 요구사항 변경을 자동으로 반영하지 않으므로, 핵심 FR이 완성된 후 현재 정책과 다시 대조해야 한다.

앞으로는 다음 순서로 진행한다.

* 운동 계획 후보 재생성 FR 완성
* 운동 세션 FR 작성
* 운동 기록 FR 작성
* 자연어 구조화 FR 작성
* 기록 조회 FR 작성
* ADR 최신화
* Domain Model 작성
* ERD와 API 설계
