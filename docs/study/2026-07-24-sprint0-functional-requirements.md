# Sprint 0 기능 요구사항(FR-PLAN-001~006) 학습

## 오늘 진행한 작업

오늘은 운동 계획(Workout Plan) 기능 요구사항의 핵심 흐름인 FR-PLAN-001부터 FR-PLAN-006까지를 작성하고 여러 차례 검토를 진행하였다.

이번 작업에서는 단순히 기능을 정의하는 것이 아니라 운동 계획 후보 생성부터 수정, 선택까지의 책임과 정책을 명확히 구분하는 데 집중하였다.

또한 ChatGPT와 Codex의 반복 리뷰를 통해 기능 간 책임 충돌, 테스트 가능성, 정책 일관성을 검토하고 요구사항을 보완하였다.

최종적으로 모든 Must Fix를 해결하고 FR-PLAN-001~006을 완료하였다.

---

# 오늘 학습한 내용

## 기능 요구사항은 시스템의 책임을 정의하는 문서이다.

기능 요구사항은 화면에서 어떤 버튼을 누르는지가 아니라 시스템이 어떤 조건에서 어떤 동작을 수행하고 어떤 결과를 제공해야 하는지를 정의하는 문서이다.

사용자 스토리가 사용자의 가치와 목적을 표현한다면 기능 요구사항은 이를 시스템 동작으로 구체화한다.

---

## Business Rule과 구현 방법은 다르다.

Business Rule은 서비스가 반드시 지켜야 하는 정책이다.

예를 들어

* 후보 수정은 후보 재생성을 발생시키지 않는다.
* 후보 수정만으로 운동 계획을 확정하지 않는다.

는 Business Rule이다.

반면

* PATCH API 사용
* Redis 사용
* Drag & Drop UI

등은 구현 방법이므로 기능 요구사항에서 확정하지 않는다.

---

## Acceptance Criteria는 테스트 기준이다.

Acceptance Criteria는 구현이 완료되었는지 판단하기 위한 테스트 기준이다.

정상 흐름뿐 아니라

* 잘못된 입력
* 실패
* 취소
* 중복 요청

까지 테스트 가능한 형태로 정의해야 한다.

---

## 기능 간 책임을 명확히 구분해야 한다.

이번 작업에서는 다음 기능들의 책임을 명확하게 분리하였다.

* 운동 계획 후보 생성
* 운동 계획 후보 수정
* 운동 계획 후보 재생성
* 운동 계획 후보 선택
* 운동 계획 후보 거절
* 운동 세션 시작

사용자 입장에서는 비슷해 보일 수 있지만 시스템에서는 각각 다른 Command라는 점을 이해하였다.

---

## Edit와 Regenerate는 다른 기능이다.

운동 종목을 삭제하거나 순서를 변경하는 것은 기존 후보를 수정(Edit)하는 것이다.

반면

* 운동 가능 시간 변경
* 운동 유형 변경
* 사용 가능한 장비 변경

은 새로운 추천 조건이므로 Regenerate에 해당한다.

이 둘을 구분하지 않으면 후보 수정 기능 안에서 AI를 다시 호출하는 문제가 발생할 수 있다.

# AI Love Workout에 적용한 내용

이번 기능 요구사항을 통해 운동 계획의 전체 흐름을 다음과 같이 정리하였다.

```
운동 계획 후보 생성
        ↓
후보 확인
        ↓
선택 / 수정 / 거절
        ↓
확정 운동 계획
        ↓
운동 세션 시작
```

또한 운동 계획 후보와 확정 운동 계획, 운동 세션을 서로 다른 개념으로 구분하였다.

---

# AI Love Workout에 적용하며 이해한 점

* 운동 계획 후보와 확정 운동 계획은 서로 다른 개념이라는 점을 이해하였다.
* 후보 수정과 후보 재생성은 시스템 책임이 다르므로 별도 기능으로 설계해야 함을 이해하였다.
* 기능 요구사항에서는 구현 기술보다 정책과 테스트 가능한 동작을 먼저 정의해야 한다는 점을 이해하였다.

---

# Codex 리뷰에서 배운 점

이번 작업에서는 여러 차례 Codex 리뷰를 진행하였다.

가장 크게 배운 점은 Recommendation과 Must Fix를 구분하는 기준이었다.

Must Fix는 단순히 더 좋은 문장을 제안하는 것이 아니라

* 정책 충돌
* 테스트 불가능
* 구현 결과가 달라질 수 있는 요구사항
* 필수 기능 누락

처럼 구현 결과를 바꾸는 문제여야 한다.

또한 기능 요구사항에서는 구현 기술보다 시스템 정책과 책임을 먼저 정의해야 한다는 점을 다시 확인하였다.

---

# 오늘의 한 문장

기능 요구사항은 기능을 설명하는 문서가 아니라, 시스템의 책임과 정책을 테스트 가능한 형태로 정의하는 문서이다.
