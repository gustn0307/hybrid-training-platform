# Sprint 0 기능 요구사항과 운동 계획 후보 수정 학습

## 오늘 진행한 작업

오늘은 기능 요구사항 문서를 작성하는 과정에서 `FR-PLAN-006 운동 계획 후보 수정`의 범위와 시스템 책임을 검토했다.

운동 계획 후보 수정 기능을 단순히 “사용자가 운동 계획을 수정할 수 있다”라고 정의하지 않고 다음 기능과의 차이를 구분했다.

* 기존 운동 계획 후보 수정
* 새로운 운동 계획 후보 재생성
* 수정된 후보 선택 및 확정 운동 계획 생성
* 운동 계획 후보 거절
* 확정 운동 계획을 이용한 운동 세션 시작

이번 학습의 핵심은 운동 계획 후보에서 어떤 값을 수정할 수 있는지, 수정 후에도 기존 후보가 동일한 후보로 유지되는지, AI의 최초 추천 이유를 어떻게 처리할지를 결정하는 것이었다.

## 오늘 학습한 내용

### 기능 요구사항

내가 이해한 의미:

기능 요구사항은 사용자 스토리에서 정의한 사용자 가치를 실제 시스템이 제공해야 하는 동작과 규칙으로 구체화한 것이다.

사용자 스토리는 사용자가 무엇을 왜 원하는지를 사용자 관점에서 표현한다.

기능 요구사항은 해당 가치를 제공하기 위해 시스템이 어떤 조건에서 어떤 동작을 수행하고 어떤 결과를 제공해야 하는지를 정의한다.

예를 들어 사용자 스토리는 다음과 같이 표현할 수 있다.

> 사용자는 AI가 생성한 운동 계획 후보를 자신의 상황에 맞게 수정하고 싶다. 왜냐하면 AI 추천을 참고하면서도 최종 운동 계획은 자신이 결정하고 싶기 때문이다.

이 사용자 스토리는 기능 요구사항에서 다음과 같이 구체화될 수 있다.

* 사용자는 운동 계획 후보의 운동 종목을 변경할 수 있어야 한다.
* 사용자는 운동 종목을 추가하거나 삭제할 수 있어야 한다.
* 사용자는 운동 종목의 순서를 변경할 수 있어야 한다.
* 사용자는 중량, 반복 횟수, 세트 수와 수행 시간을 변경할 수 있어야 한다.
* 사용자는 러닝 목표 페이스와 자유 러닝 여부를 변경할 수 있어야 한다.
* 사용자는 운동 유형별 배정 시간을 변경할 수 있어야 한다.
* 후보 수정은 새로운 AI 추천 후보 생성을 발생시키지 않아야 한다.
* 후보 수정만으로 확정 운동 계획을 생성해서는 안 된다.
* 후보 수정만으로 운동 세션을 시작해서는 안 된다.

#### 피드백 및 보완할 부분

기능 요구사항은 단순한 화면 기능 목록이 아니다.

다음 내용을 함께 정의해야 한다.

* 기능을 수행할 수 있는 사용자
* 기능을 수행할 수 있는 상태
* 시스템이 제공해야 하는 동작
* 수정 가능한 값
* 수정할 수 없는 값
* 정상 처리 결과
* 잘못된 입력이나 실패 시 처리
* 다른 기능과의 책임 경계
* 테스트 가능한 완료 기준

“운동 계획을 수정할 수 있다”라고만 작성하면 어떤 값을 수정할 수 있는지, AI를 다시 호출하는지, 수정과 동시에 계획이 확정되는지를 판단할 수 없다.

따라서 수정 가능한 항목과 수정, 재생성, 확정, 세션 시작의 책임을 명확히 분리해야 한다.

내가 이해한 의미:

사용자 스토리는 사용자 관점에서 원하는 행동과 그 행동을 통해 얻고 싶은 가치를 표현한다.

기능 요구사항은 그 사용자 가치를 제공하기 위해 시스템이 수행해야 하는 구체적인 동작과 정책을 표현한다.

예를 들어 사용자 스토리가 다음과 같을 수 있다.

이에 연결되는 기능 요구사항은 여러 개로 나뉠 수 있다.

* 운동 종목 변경
* 운동 종목 추가
* 운동 종목 삭제
* 운동 종목 순서 변경
* 중량 변경
* 반복 횟수 변경
* 세트 수 변경
* 수행 시간 변경

* 자유 러닝 여부 변경
* 운동 유형별 배정 시간 변경
* 수정값 유효성 검증
* 수정 후 예상 소요 시간 처리
* AI 최초 추천 이유 유지

#### 피드백 및 보완할 부분

사용자 스토리 하나가 반드시 하나의 기능 요구사항으로만 연결되는 것은 아니다.

하나의 사용자 스토리는 여러 기능 요구사항으로 구체화될 수 있다.

사용자 스토리는 구현 방식보다 사용자 가치를 중심으로 작성하고, 기능 요구사항은 개발과 테스트가 가능할 정도로 시스템 동작을 구체화해야 한다.

### Command와 Query

내가 이해한 의미:

Query는 시스템의 상태를 변경하지 않고 데이터를 읽는 요청이다.

Command는 시스템의 상태를 변경하는 요청이다.

AI Love Workout에서 Query에 해당할 수 있는 기능은 다음과 같다.

* 운동 계획 후보 조회
* 운동 세션 조회
* 운동 기록 조회
* 날짜별 운동 내역 조회

Command에 해당할 수 있는 기능은 다음과 같다.

* 운동 계획 후보 생성
* 운동 계획 후보 수정
* 운동 계획 후보 거절
* 확정 운동 계획 생성
* 운동 세션 시작
* 운동 기록 저장
* 운동 세션 종료

`FR-PLAN-006 운동 계획 후보 수정`은 기존 후보의 상태를 변경하므로 Command에 해당한다.

#### 피드백 및 보완할 부분

Command와 Query의 차이는 HTTP 메서드 자체가 아니라 시스템의 상태를 변경하는지에 있다.

Command를 설계할 때는 다음 사항도 검토해야 한다.

* 누가 변경할 수 있는가
* 어떤 상태에서 변경할 수 있는가
* 어떤 값까지 변경할 수 있는가
* 변경 결과가 어디까지 영향을 주는가
* 요청이 실패했을 때 기존 상태를 유지하는가
* 같은 요청이 반복되었을 때 어떻게 처리하는가

이번 후보 수정 기능에서는 수정 가능한 필드, 후보의 정체성 유지와 수정 실패 시 기존 유효한 상태 보존이 중요한 설계 기준이다.

### Edit와 Regenerate

내가 이해한 의미:

Edit는 AI가 이미 생성한 기존 운동 계획 후보의 일부 내용을 사용자가 직접 변경하는 것이다.

Regenerate는 AI에 다시 추천을 요청하여 새로운 운동 계획 후보를 생성하는 것이다.

Edit의 흐름은 다음과 같다.

```
AI가 운동 계획 후보 A 생성
→ 사용자가 후보 A의 일부 내용 변경
→ 수정된 운동 계획 후보 A
```

Regenerate의 흐름은 다음과 같다.

```
사용자가 추천 생성 조건 변경 또는 재생성 요청
→ AI가 다시 운동 계획 생성
→ 새로운 운동 계획 후보 B
```

#### 수정에 포함하기로 결정한 항목

* 운동 종목 변경
* 운동 종목 추가
* 운동 종목 삭제
* 운동 종목 순서 변경
* 중량 변경
* 반복 횟수 변경
* 세트 수 변경
* 수행 시간 변경
* 러닝 목표 페이스 변경
* 자유 러닝 여부 변경
* 운동 유형별 배정 시간 변경

#### 재생성 입력으로 구분한 항목

* 오늘 운동 가능한 시간
* 선택한 운동 유형
* 사용 가능한 장비

#### MVP 수정 대상에서 제외한 항목

* 휴식 시간

#### 피드백 및 보완할 부분

수정과 재생성은 사용자 화면에서는 비슷해 보일 수 있지만 시스템 책임은 다르다.

운동 종목 하나를 삭제하는 것은 기존 후보를 사용자가 직접 편집한 것이므로 Edit이다.

반면 오늘 운동 가능한 시간을 30분에서 60분으로 변경하면 운동 종목 수, 강도와 운동 유형별 시간 배분을 다시 계산해야 할 수 있다.

따라서 운동 가능 시간 변경은 기존 후보의 단순 수정이 아니라 AI가 새로운 후보를 생성하는 Regenerate로 분리하는 것이 적절하다.

이 구분이 없으면 수정 기능 안에서 AI를 다시 호출하거나 새로운 후보를 기존 후보의 수정 결과처럼 처리하는 문제가 발생할 수 있다.

### 운동 계획 후보의 Identity

내가 이해한 의미:

Identity는 객체의 일부 속성이 변경되더라도 해당 객체가 계속 같은 객체인지 판단하는 정체성이다.

예를 들어 AI가 다음 후보를 생성했다고 가정한다.

```
후보 A

1. 푸시업
2. 스쿼트
3. 러닝
```

사용자가 푸시업을 삭제하면 다음과 같이 변경된다.

```
후보 A

1. 스쿼트
2. 러닝
```

운동 종목 목록은 변경되었지만 사용자가 AI에게 새로운 후보를 요청한 것은 아니다.

따라서 이는 새로운 후보 B가 아니라 기존 후보 A의 수정된 상태이다.

#### 내가 직접 답한 내용

운동 종목 하나를 삭제하는 것은 Edit이다.

그 이유는 주어진 운동 계획 후보에서 일부 내용을 편집한 것이기 때문이다.

#### 피드백 및 보완할 부분

객체의 Identity와 객체의 현재 상태는 구분해야 한다.

운동 종목, 순서, 세트 수와 같은 속성이 바뀌면 후보의 상태는 달라진다.

하지만 새로운 후보를 생성한 것이 아니라면 후보를 식별하는 Identity까지 반드시 달라지는 것은 아니다.

이 구분은 추후 Domain Model과 API 설계에 영향을 준다.

기존 후보를 수정하는 기능은 개념적으로 다음과 연결될 수 있다.

```
PATCH /workout-plan-candidates/{candidateId}
```

새로운 후보를 생성하거나 재생성하는 기능은 별도의 Command로 연결될 수 있다.

다만 구체적인 URL과 HTTP 메서드는 API 설계 단계에서 확정한다.

### 운동 종목 삭제

내가 이해한 의미:

운동 종목 하나를 삭제하는 것은 운동 계획 후보 전체를 거절한 것이 아니다.

사용자는 AI가 추천한 계획을 참고하면서 자신에게 필요하지 않거나 수행하기 어려운 종목만 제거한 것이다.

따라서 다음 행동은 서로 구분해야 한다.

* 후보에서 특정 운동 종목 하나 삭제
* 후보에 새로운 운동 종목 추가
* 후보의 모든 운동 종목 삭제
* 운동 계획 후보 전체 거절
* 후보를 선택하지 않고 흐름 종료

운동 종목 하나의 삭제는 후보 수정에 해당한다.

#### 아직 결정되지 않은 부분

후보에 포함된 모든 운동 종목을 삭제하는 것을 허용할지는 아직 확정하지 않았다.

다음 정책을 비교해야 한다.

* 모든 운동 종목 삭제 자체를 허용하지 않는다.
* 편집 중에는 허용하지만 유효하지 않은 후보는 확정할 수 없게 한다.
* 모든 운동 종목을 삭제하면 후보 거절로 처리한다.

모든 운동 종목 삭제와 후보 거절은 사용자의 의도가 다를 수 있으므로 자동으로 같은 행동으로 처리하지 않는 방향이 적절해 보인다.

최종 정책은 `FR-PLAN-006 후보 수정`과 `FR-PLAN-007 후보 거절`의 책임을 비교한 뒤 결정해야 한다.

### 운동 종목 순서 변경

내가 이해한 의미:

사용자는 AI가 추천한 운동 종목 자체에는 동의하더라도 자신의 운동 습관, 운동 장소 이동이나 선호에 따라 운동 순서를 바꾸고 싶을 수 있다.

예를 들어 AI가 다음 순서를 추천할 수 있다.

```
1. 러닝
2. 푸시업
3. 스쿼트
```

사용자는 다음과 같이 변경할 수 있다.

```
1. 푸시업
2. 스쿼트
3. 러닝
```

이는 새로운 추천 조건을 입력한 것이 아니라 기존 후보의 내용을 편집한 것이므로 Edit에 해당한다.

#### 오늘 결정한 내용

* 운동 종목 순서는 수정 가능 항목에 포함한다.
* 수정된 운동 순서는 사용자가 확정하는 운동 계획에 반영한다.
* 계획 단계의 운동 순서와 실제 운동 세션에서 수행한 순서는 별개의 개념이다.
* 운동 세션에서는 사용자가 계획과 다른 순서로 운동할 수도 있다.
* 기능 요구사항에서는 드래그 앤 드롭이나 위·아래 버튼과 같은 구체적인 UI 방식을 확정하지 않는다.

#### 피드백 및 보완할 부분

기능 요구사항은 사용자가 운동 순서를 변경할 수 있어야 한다는 시스템 동작을 정의한다.

구체적인 조작 방법은 UI 설계 단계에서 결정한다.

또한 계획 순서를 수정할 수 있다고 해서 실제 운동 수행 순서를 시스템이 강제해야 하는 것은 아니다.

### AI 추천 이유와 사용자 수정

내가 이해한 의미:

AI의 추천 이유는 사용자의 최근 운동 기록, 운동 가능 시간, 선택한 운동 유형과 사용 가능한 장비 등의 데이터를 바탕으로 최초 운동 계획 후보를 생성한 근거이다.

사용자가 운동 계획 후보를 수정하더라도 AI가 최초 추천을 생성할 때 사용한 데이터와 판단이 사라지는 것은 아니다.

추천 이유를 확인한 뒤에도 운동 계획을 수정한 것은 사용자의 결정이므로 최초 추천 이유를 삭제할 필요는 없다고 판단했다.

#### 내가 직접 답한 내용

운동 계획을 수정하더라도 AI의 추천 이유는 데이터를 바탕으로 추천한 것이기 때문에 사라지는 것은 아니라고 생각했다.

추천 이유를 들은 뒤 해당 운동 계획을 수정한 것은 사용자이므로 추천 이유를 삭제할 필요는 없다고 판단했다.

#### 피드백 및 보완할 부분

AI의 최초 추천 이유와 수정된 현재 계획 전체에 대한 설명은 같지 않을 수 있다.

예를 들어 AI가 최근 상체 운동 기록이 부족하다는 이유로 푸시업을 추천했지만 사용자가 푸시업을 삭제할 수 있다.

이 경우 다음 추천 근거는 여전히 사실이다.

> 최근 상체 운동 기록이 부족하여 푸시업을 추천했다.

하지만 이 이유를 수정된 현재 계획 전체의 추천 이유라고 표시하면 사용자가 혼동할 수 있다.

따라서 다음 책임을 구분해야 한다.

* AI는 최초 운동 계획 후보와 최초 추천 이유를 생성한다.
* 사용자는 추천 이유를 확인하고 후보를 수정할 수 있다.
* 시스템은 사용자 수정 내용을 반영하면서 최초 추천 이유를 유지한다.
* 최초 추천 이유를 수정된 전체 계획에 대한 새로운 AI 설명으로 취급하지 않는다.
* 사용자 수정 내용을 AI가 추천한 내용처럼 표현하지 않는다.
* 후보 수정 시 AI 추천 이유를 자동으로 다시 생성하지 않는다.

표현 예시는 다음과 같다.

```
AI 최초 추천 이유

최근 상체 운동 기록이 부족하여
푸시업을 포함했습니다.
```

추천 이유를 화면에서 구체적으로 어떻게 보여줄지는 UI 설계 단계에서 결정한다.

---

### 후보 수정 시 추천 이유를 다시 생성하지 않는 이유

내가 이해한 의미:

운동 계획 후보를 수정할 때마다 AI 추천 이유를 다시 생성하면 단순한 Edit 기능 안에서 AI 호출이 다시 발생한다.

```
사용자 후보 수정
→ AI 재호출
→ 수정된 계획 분석
→ 새로운 이유 생성
```

이렇게 되면 Edit와 Regenerate의 책임이 불명확해질 수 있다.

#### 추천 이유를 다시 생성하지 않을 때의 장점

* Edit와 Regenerate의 책임을 명확히 유지할 수 있다.
* 불필요한 AI 호출 비용을 줄일 수 있다.
* 후보 수정 시 응답 지연을 줄일 수 있다.
* AI가 사용자의 수정 의도를 잘못 해석해 사실과 다른 이유를 생성하는 것을 방지할 수 있다.
* AI의 최초 제안과 사용자의 최종 결정을 구분할 수 있다.
* 사용자가 어떤 AI 추천을 보고 어떤 결정을 내렸는지 설명 가능성을 유지할 수 있다.

#### 향후 확장 가능성

향후 사용자가 명시적으로 요청하면 수정된 운동 계획을 AI가 다시 검토하고 별도의 의견을 제공하는 기능을 고려할 수 있다.

다만 이 기능은 최초 추천 이유를 자동으로 변경하는 것이 아니라 별도의 AI 검토 기능으로 설계하는 것이 적절하다.

### Human-in-the-loop

내가 이해한 의미:

Human-in-the-loop는 AI가 생성한 결과를 자동으로 최종 확정하지 않고, 사용자가 결과를 확인하고 수정하거나 거절한 뒤 최종 결정을 내리도록 하는 설계 방식이다.

AI Love Workout에서는 다음 흐름으로 적용된다.

```
사용자가 추천 생성 조건 입력
→ AI가 운동 계획 후보 생성
→ 사용자가 후보와 추천 이유 확인
→ 사용자가 후보 선택·수정·거절
→ 사용자 확인 후 확정 운동 계획 생성
→ 확정 운동 계획을 기반으로 운동 세션 시작
```

AI는 사용자의 결정을 돕는 후보를 제공하지만 최종 운동 계획을 결정하는 주체는 사용자이다.

#### 피드백 및 보완할 부분

Human-in-the-loop는 단순히 확인 버튼을 추가하는 것을 의미하지 않는다.

사용자가 AI 결과를 실질적으로 통제할 수 있어야 한다.

* AI 결과를 확인할 수 있어야 한다.
* AI 추천 이유를 확인할 수 있어야 한다.
* 잘못되거나 원하지 않는 결과를 수정할 수 있어야 한다.
* 후보 전체를 거절할 수 있어야 한다.
* 사용자 확인 전에는 최종 운동 계획으로 확정되지 않아야 한다.
* 사용자의 수정 내용과 AI가 생성한 원본 내용을 구분할 수 있어야 한다.

이번 후보 수정 정책은 AI의 추천을 참고하면서도 최종 통제권을 사용자에게 두는 Human-in-the-loop 원칙을 구체화한 것이다.

### Business Rule

내가 이해한 의미:

Business Rule은 기능이 동작할 때 반드시 지켜야 하는 서비스의 정책과 제약 조건이다.

기능 요구사항이 시스템이 무엇을 해야 하는지를 정의한다면 Business Rule은 해당 기능이 어떤 원칙과 조건을 따라야 하는지를 정의한다.

`FR-PLAN-006 운동 계획 후보 수정`에서 고려할 수 있는 Business Rule은 다음과 같다.

* 수정은 기존 운동 계획 후보를 대상으로 수행한다.
* 후보 수정은 새로운 AI 후보 생성을 발생시키지 않는다.
* 후보 수정은 AI 후보 재생성을 발생시키지 않는다.
* 후보 수정만으로 확정 운동 계획을 생성하지 않는다.
* 후보 수정만으로 운동 세션을 시작하지 않는다.
* 수정된 후보는 사용자가 별도로 확정하기 전까지 확정 운동 계획이 아니다.
* 수정 후에도 AI의 최초 추천 이유를 유지한다.
* 최초 추천 이유를 수정된 전체 계획에 대한 새로운 AI 설명으로 취급하지 않는다.
* 수정 가능한 값과 재생성이 필요한 입력 조건을 구분한다.
* 유효하지 않은 후보는 확정할 수 없어야 한다.

#### 피드백 및 보완할 부분

Business Rule은 특정 UI나 기술 구현에 종속되지 않아야 한다.

“수정 버튼을 누르면 모달을 띄운다”는 UI 방식이다.

“수정된 후보는 사용자가 별도로 확정하기 전까지 확정 운동 계획이 아니다”는 서비스 정책이므로 Business Rule이다.

Business Rule은 이후 Domain Model, API, 데이터베이스와 테스트 설계 전체에 영향을 줄 수 있다.

### Acceptance Criteria

내가 이해한 의미:

Acceptance Criteria는 기능 요구사항이 어떤 조건에서 완료된 것으로 판단되는지를 테스트 가능한 형태로 정의한 기준이다.

운동 계획 후보 수정에서는 정상적인 수정뿐 아니라 잘못된 입력, 수정 실패와 다른 기능의 잘못된 실행 여부도 확인해야 한다.

예시는 다음과 같다.

```
Given 사용자가 AI가 생성한 운동 계획 후보를 확인하고 있을 때
When 운동 종목 하나를 삭제하면
Then 해당 운동 종목이 기존 후보에서 제거되어야 한다.
And 새로운 운동 계획 후보를 생성해서는 안 된다.
And 운동 계획을 자동으로 확정해서는 안 된다.
And 운동 세션을 자동으로 시작해서는 안 된다.
```

```
Given 사용자가 운동 계획 후보를 확인하고 있을 때
When 운동 종목의 순서를 변경하면
Then 변경된 순서가 해당 후보에 반영되어야 한다.
And 해당 후보를 확정할 경우 변경된 순서가 확정 운동 계획에 반영되어야 한다.
```

```
Given 사용자가 운동 계획 후보의 세트 수를 수정할 때
When 허용된 범위의 값을 입력하면
Then 수정된 세트 수가 기존 후보에 반영되어야 한다.
```

```
Given 사용자가 운동 계획 후보를 수정할 때
When 허용되지 않는 값을 입력하면
Then 해당 수정 요청을 완료해서는 안 된다.
And 사용자가 수정해야 할 값을 확인할 수 있어야 한다.
And 수정 전 유효한 후보 상태가 손상되어서는 안 된다.
```

```
Given 사용자가 AI 추천 이유가 포함된 운동 계획 후보를 수정할 때
When 후보의 운동 종목이나 수행 값을 변경하면
Then AI의 최초 추천 이유는 유지되어야 한다.
And 수정된 계획 전체에 대한 새로운 AI 추천 이유를 자동 생성해서는 안 된다.
```

#### 피드백 및 보완할 부분

인수 조건은 “수정이 성공한다”는 결과만 확인해서는 부족하다.

다음 사항도 함께 검증해야 한다.

* 기존 후보의 Identity가 유지되는가
* 후보 재생성이 발생하지 않는가
* 자동 운동 계획 확정이 발생하지 않는가
* 자동 운동 세션 시작이 발생하지 않는가
* 잘못된 값이 반영되지 않는가
* 실패 시 기존 유효한 상태가 유지되는가
* 수정된 값에 따라 예상 소요 시간이 올바르게 처리되는가
* AI의 최초 추천 이유가 유지되는가
* 사용자 수정 결과를 AI의 새로운 추천으로 표현하지 않는가

## AI Love Workout에 적용할 내용

* 운동 계획 후보 수정과 운동 계획 후보 재생성을 서로 다른 기능으로 구분한다.
* 후보 수정은 기존 운동 계획 후보의 Identity를 유지한다.
* 사용자는 후보의 운동 종목을 변경, 추가 또는 삭제할 수 있다.
* 사용자는 운동 종목의 순서를 변경할 수 있다.
* 사용자는 운동별 중량, 반복 횟수, 세트 수와 수행 시간을 수정할 수 있다.
* 사용자는 러닝 목표 페이스와 자유 러닝 여부를 수정할 수 있다.
* 사용자는 운동 유형별 배정 시간을 수정할 수 있다.
* 운동 가능 시간, 선택한 운동 유형과 사용 가능한 장비 변경은 후보 수정이 아니라 재생성 입력 변경으로 처리한다.
* 휴식 시간은 MVP 후보 수정 대상에 포함하지 않는다.
* 후보 수정은 새로운 AI 후보를 생성하지 않는다.
* 후보 수정은 확정 운동 계획을 자동으로 생성하지 않는다.
* 후보 수정은 운동 세션을 자동으로 시작하지 않는다.
* 수정된 후보는 사용자가 별도로 확정하기 전까지 확정 운동 계획이 아니다.
* AI의 최초 추천 이유는 후보를 수정해도 유지한다.
* 최초 추천 이유는 수정된 현재 계획 전체에 대한 새로운 설명으로 취급하지 않는다.
* 후보 수정 시 AI 추천 이유를 자동으로 다시 생성하지 않는다.
* 사용자 수정 내용과 AI가 최초 생성한 내용을 구분할 수 있는 구조를 고려한다.
* 수정 후 예상 소요 시간과 운동 유형별 배정 시간의 정합성을 검증해야 한다.
* 유효하지 않은 후보는 확정할 수 없도록 검증해야 한다.

## 오늘 내가 직접 결정한 것

* 운동 종목 하나를 삭제하는 것은 기존 운동 계획 후보를 편집한 것이므로 Edit로 처리한다.
* 운동 종목을 삭제해도 기존 후보의 Identity는 유지한다.
* 운동 종목 삭제로 인해 새로운 AI 운동 계획 후보를 생성하지 않는다.
* 운동 종목 순서는 사용자가 수정할 수 있어야 한다.
* 수정된 운동 종목 순서는 확정 운동 계획에 반영한다.
* 계획 단계의 운동 순서와 실제 세션에서 수행한 순서는 구분한다.
* 운동 계획 후보를 수정해도 AI의 최초 추천 이유는 유지한다.
* AI 추천 이유는 최초 데이터를 바탕으로 생성된 판단이므로 사용자 수정으로 인해 자동 삭제하지 않는다.
* 사용자는 AI의 추천 이유를 확인한 뒤에도 자신의 판단으로 후보를 수정할 수 있다.
* AI 최초 추천 이유와 사용자 수정 결과의 책임을 구분한다.
* 후보 수정 시 AI 추천 이유를 자동으로 다시 생성하지 않는다.
* 후보 수정은 후보 재생성, 확정 운동 계획 생성과 운동 세션 시작을 발생시키지 않는다.

* 수정 완료 시 한 번에 저장할지
* 후보를 확정할 때만 저장할지
* 운동 유형별 배정 시간을 변경하면 개별 운동 시간이 자동으로 조정되는지
* 수정된 운동 종목과 직접 연결된 AI 추천 이유를 UI에서 어떻게 표시할지
* 수정 실패 시 전체 요청을 취소할지
* 유효한 일부 변경만 반영할지
* 수정된 후보의 예상 소요 시간을 어떤 규칙으로 계산할지
* 운동 순서 변경에 안전성 검증이 필요한지

#### 피드백 및 보완할 부분

기능 요구사항 단계에서는 기술 구현을 너무 일찍 확정하지 않아야 한다.

예를 들어 다음 항목은 기능 요구사항에서 사용자 가치와 시스템 규칙을 먼저 결정한 뒤 이후 설계 단계에서 기술적으로 확정해야 한다.

* Redis 사용 여부
* 원본 후보 저장 테이블 구조
* 수정 이력 테이블 구조
* 즉시 저장 또는 일괄 저장
* API 요청 형식
* 프론트엔드 상태 관리 방식

먼저 다음 질문에 답해야 한다.

* 수정한 내용이 유실되지 않아야 하는가
* 사용자가 원본 AI 추천을 다시 확인할 필요가 있는가
* 수정 사실을 추후 분석하거나 설명할 필요가 있는가
* 오류 발생 시 어떤 상태를 유지해야 하는가

이 요구를 결정한 뒤 Domain Model, API와 데이터베이스 설계 단계에서 구현 방식을 선택한다.

## 기능 요구사항이 이후 설계에 미치는 영향

### Domain Model

운동 계획 후보를 표현하는 도메인 객체가 필요하다.

예시:

```
WorkoutPlanCandidate
```

운동 계획 후보는 여러 운동 항목을 포함할 수 있다.

```
WorkoutPlanCandidate
└── WorkoutPlanItem
```

운동 항목에는 다음 값이 포함될 수 있다.

* 운동 종목
* 운동 순서
* 중량
* 반복 횟수
* 세트 수
* 수행 시간
* 러닝 목표 페이스
* 자유 러닝 여부

운동 계획 후보에는 AI가 생성한 정보와 사용자가 수정한 현재 상태를 구분할 필요가 생길 수 있다.

구체적인 객체 구조와 책임은 Domain Model 설계 단계에서 확정한다.

### API

후보 수정은 기존 후보의 상태를 변경하는 Command이다.

개념적으로 다음과 같은 형태를 고려할 수 있다.

```
PATCH /workout-plan-candidates/{candidateId}
```

다만 API 설계 전에 다음 정책을 먼저 결정해야 한다.

* 부분 수정이 가능한가
* 여러 운동 항목을 한 번에 수정할 수 있는가
* 수정 값의 유효성 검증 기준은 무엇인가
* 수정 실패 시 기존 상태를 유지하는가
* 후보 수정과 후보 확정을 별도 API로 분리하는가
* 후보 수정과 후보 재생성을 별도 API로 분리하는가

### 데이터베이스

후보 수정 기능으로 인해 다음 데이터를 저장할 수 있다.

* 후보 식별자
* 후보의 현재 상태
* 후보에 포함된 운동 항목
* 운동 항목의 순서
* AI 최초 추천 이유
* 사용자 수정 여부
* 생성 시각
* 수정 시각

원본 후보와 수정 이력까지 저장할지는 추가 요구사항 검토 후 결정한다.

### 테스트

다음과 같은 테스트가 필요할 수 있다.

* 운동 종목 변경 성공
* 운동 종목 추가 성공
* 운동 종목 삭제 성공
* 운동 종목 순서 변경 성공
* 중량 변경 성공
* 반복 횟수 변경 성공
* 세트 수 변경 성공
* 수행 시간 변경 성공
* 러닝 목표 페이스 변경 성공
* 자유 러닝 여부 변경 성공
* 유효하지 않은 값의 수정 실패
* 수정 실패 시 기존 후보 유지
* 후보 수정 시 AI 재생성 미발생
* 후보 수정 시 자동 확정 운동 계획 생성 미발생
* 후보 수정 시 자동 세션 시작 미발생
* 후보 수정 후 AI 최초 추천 이유 유지
* 수정 후 예상 소요 시간 검증

## 기능 요구사항 작성에서 주의할 점

* 사용자 요구와 기술 구현 방법을 구분한다.
* “드래그 앤 드롭으로 순서를 변경한다”보다 “사용자는 운동 순서를 변경할 수 있어야 한다”라고 작성한다.
* “Redis에 수정 내용을 저장한다”보다 수정 내용의 유실 방지 필요성을 먼저 정의한다.
* 수정 가능한 필드를 “운동 정보를 수정할 수 있다”처럼 추상적으로 쓰지 않고 구체적으로 나열한다.
* 수정, 재생성, 확정 운동 계획 생성과 세션 시작의 책임을 섞지 않는다.
* 정상 흐름뿐 아니라 잘못된 값, 취소와 처리 실패도 정의한다.
* Acceptance Criteria는 실제 테스트가 가능한 관찰 가능한 결과를 사용한다.
* 상위 MVP 범위와 사용자 스토리에서 기능의 근거를 확인한다.
* 아직 결정되지 않은 정책을 확정된 것처럼 작성하지 않는다.
* FR에서 상위 문서에 없는 Must 기능을 임의로 추가하지 않는다.

## 오늘의 한 문장

운동 계획 후보 수정은 AI가 생성한 기존 후보의 Identity를 유지하면서 사용자가 후보의 일부 상태를 변경하는 Command이며, 새로운 후보를 생성하는 재생성 및 사용자 확인이 필요한 확정 운동 계획 생성과 책임을 분리해야 한다.

## 개인적인 느낌

처음에는 운동 종목 삭제와 순서 변경이 단순한 화면 편집 기능이라고 생각할 수 있었다.

하지만 수정과 재생성을 구분하는 과정에서 객체의 Identity, AI 호출 여부, 추천 이유 보존, Human-in-the-loop, API와 Domain Model까지 연결된다는 점을 알게 되었다.

기능 하나를 정의하는 데도 생각해야 할 정책이 많아 시간이 오래 걸리지만, 구현 전에 책임을 나누지 않으면 후보 수정 API가 AI를 다시 호출하거나 사용자 수정 결과를 AI 추천으로 표현하는 등 기능이 뒤섞일 수 있다는 점을 이해했다.
